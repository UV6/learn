## SPEC 模式
三个文件：
- spec.md 项目范围：在做什么，为什么要做，范围是什么
- tasks：任务拆解：把 spec 具体化成几个模块，每个模块有几个子任务
- checklist 验证清单：ai 完成之后，逐项过一遍这分份清单，确认没有遗漏
```Markdown
我正在构建一个终端 AI 编程助手（类似 Claude Code），项目名叫 MewCode，使用的编程语言是java。

每次我会提出一个初步的想法，需要你通过向我提问，帮助我澄清需求、挖掘边缘场景。澄清清楚后共创三份文档保存到项目根目录：

# 三份文档的角色与边界

## spec.md
回答：要解决什么问题、做哪些能力、不做哪些、什么算完成。
写：背景、目标用户、能力清单（一句话一条）、非功能要求、设计骨架、Out of Scope
不写：具体函数名 / 参数名 / 默认值 / 错误文本 / 行号 / SDK 类型名
   （这些是实现细节，spec 改一次就过期，维护爆炸）

## tasks.md
回答：按什么顺序做、每步动什么文件。
- 5~15 个任务，每个能在一次专注会话内完成
- 每个任务标注：影响文件、依赖任务、参考资料定位（精确到函数/行号都可以）
- 最后一定有「接入主流程」+「端到端验证」两个任务

## checklist.md
每一项必须可勾选、可观测，不许写「实现完整」「质量良好」。
- 把 spec 里被砍掉的具体值（错误文本、默认值、阈值）放进来作为验收项
- 写法举例：「`grep -r X` 返回 ≥3 条」「输入 Y 看到输出 Z」
- 至少一条端到端验收
  
# plan.md
  怎么做
  架构概览、组件划分、核心接口与数据结构、模块交互、技术决策
```

## 流式输出
本质上是一个长连接的 http 响应，服务器往里面持续写数据

## token
输出是输入的 5 倍价格
一个英文单词是 1-2 个 token，一个汉字是 1-2 个 token。
会话历史越长，输入越贵，每轮请求都会把完整的对话历史作为输入发过去

## 封装层
消息类型、事件解析、错误处理可以随时切换不同的协议。需要在 LLM 客户端做一层封装。配置上只要四个字段就能覆盖所有主流供应商。protocol 决定走哪家的 api 协议，model 指定模型，base_url 指定端点地址，apikey 做认证。
所以为了做适配，在客户端要根据不同的协议分发到不同的后端实现：把 message 转到供应商的格式、调用对应的 api、把供应商的事件转成自定义 StreanEvent 流事件、通过异步流返回给调用方

## 怎么从单轮到多轮？
答案很简单：每次调用 API 都把完整的对话历史发过去

## 消息模型：两层设计
对接模型厂商：role+content，但是 Agent 运行过程中，用户输入、模型回复、API 调用失败、错误信息、工具调用。只是两种不够用。而且在流式输出中，模型的回复还有状态：刚发送是 streaming，接收完是 complet，出错是 error。有了状态，UI 就能进行渲染，格式转换遇到 error 就能过滤。唯一 Id 也很关键，可以通过 id 指定消息，不断追加文本。

对接模型厂商：role+content，专门跟 LLM 通信。LLM API 是无状态的
内部层：user、assistant、system、tool。每条消息带一个唯一 Id，方便定位更新。还有时间戳、token 用量、响应耗时。关键点：状态字段。
## 对话管理器
流式接收场景：消息列表里：后台可能往一条 assistant 里追加文字，但是另一边正在读消息列表。两个地方同时操作一个列表，就要保证并发安全。
所以需要对话管理器保证并发。

### 格式转换
把内部状态变成API 能用的干净模式
要做这些事情：
- 过滤：system 角色的消息是内部概念、error 状态的 assistant 消息过滤
- 合并：自动合并相邻的同角色消息
- 保证首条消息是 user，并且出现 user/assistant 交替出现。
