## **Bubblewrap** 和 **Seatbelt**
- Bubblewrap：利用 Linux 的 namespace 等机制，为进程构造一个“假的、缩小版操作系统视图”。
- Seatbelt：利用 macOS 内核的强制访问控制规则，逐项判断某个操作是否允许。

## bubblewrap：Linux机制，windows 用 wsl2
一个很轻量的沙箱构造器。它总会创建新的 ** namespace**，然后由调用方明确决定沙箱里能看到哪些目录；还可以额外隔离 PID、网络、用户、IPC、主机名等 namespace。
seccomp 直接在系统调用层面过滤
namespace可以让不同的进程看到不同的资源，这样在沙箱中运行时，如果没有对应的路径根本访问不到，bubblewrap是开源的，能隔离文件目录、网络、进程、用户和权限、系统调用

## seatbelt：macOS 的内核访问控制沙箱
bubblewrap 主要通过改变进程看到的系统环境实现隔离：
```
这个文件不在你的 mount namespace 里
```
Seatbelt主要通过访问规则实现隔离：
```
这个文件真实存在，也可能路径可见，
但你的 sandbox profile 不允许读取它
```
seatbelt 可以控制的操作：文件 crud，网络、进程、系统配置、硬件能力、进程通信


## 面试怎么讲
往深了做的话，分三个层次：

1. 应用层就是咱们现在做的：路径白名单、危险命令正则拦截、权限确认弹窗。这层的定位是快速筛掉明显不该做的事，响应快、体验好，但安全性扛不住刻意绕过
    

2. 真正兜底的是 OS 级沙箱macOS 上可以用苹果内核自带的 seatbelt，在内核层面限制进程能碰哪些文件、能不能联网
    

Linux 上用 bubblewrap 加 seccomp，bubblewrap 通过 namespace 创建隔离环境，seccomp 直接在系统调用层面过滤，ptrace、mount 这些危险调用全禁掉跟应用层最大的区别是：应用层是你自己的代码在拦，模型通过 Bash 就能绕过去，OS 级是操作系统内核在拦，进程自己没权限解除，所以绕不了

3. 然后还有网络隔离，比如做域名白名单，或者在受限模式下直接断网，这样就算模型想 curl 外发数据也出不去。
    

咱们项目目前只做了应用层沙箱，是因为教学重心放在 Agent 架构设计上，做了取舍。不过这块后续我会补上 OS 级沙箱的支持，把 seatbelt 和 bubblewrap + seccomp 加进来，让安全体系更完整。

面试的时候你可以这样讲：项目里的路径沙箱是应用层的第一道防线，你清楚它的局限，生产环境需要 OS 级隔离兜底，然后把 seatbelt 和 bubblewrap + seccomp 的原理讲一下就行。
