## 2026-06-21

### 记忆的实际管理机制：会话 vs 记忆

**问：** Codia 的会话记忆是怎么管理的，会话存放在哪？每次会话都会创建一个 memory 吗？

**答：** 记忆**不是按会话创建**的，而是**每轮对话结束后**异步提炼。

- **记忆存放**：两级作用域——项目级 `<项目根>/memory/`，用户级 `~/.mewcode/memory/`，每级各有一个 `MEMORY.md` 索引文件
- **会话存放**：`<项目根>/sessions/` 下按 `YYYYMMDD-HHMMSS-xxxx.jsonl` 格式存储，与记忆是两套独立系统
- **触发链路**：`sendMessage()` → AgentLoop 结束 → `scheduleMemoryExtraction()` → `extractFromTurn()`
- **加载时机**：会话启动时一次性加载 `loadIndexes()`，渲染为 `memoryText` 注入 system prompt

### 为什么 memory 下积累了 80+ 条无意义记忆

**问：** memory 文件夹下有很多没意义的记忆，生成规则是什么？

**答：** 根因在 `extractKnowledgeCandidates()` 规则过于粗糙：

1. 只要 assistant 回复 > 200 字符 → 触发提取
2. 按 `。` 和 `\n` 断句，取 > 30 字符的句子，最多 3 句
3. 每轮对话稳定产生 1-3 条噪音（`ls` 输出片段、随口描述的项目结构等）
4. 去重用简单子串匹配，几乎不可能命中（每次回复内容不同）

### 记忆文件 frontmatter 字段含义

**问：** frontmatter 里的标签分别表示什么？

**答：**

| 字段 | 含义 |
|------|------|
| `id` | 唯一标识，格式 `know-{时间戳}-随机` 或 `pref-{时间戳}-随机` |
| `scope` | `project`（项目级）或 `user`（用户级） |
| `category` | `user_preference` / `correction_feedback` / `project_knowledge` / `reference_material` |
| `source_session` | 产生该记忆的会话 ID |
| `updated_at` | 最后更新时间 |

其中 `correction_feedback` 和 `reference_material` 定义了但代码中从未使用。

### 改进方案：LLM + Tool Call 替换关键词提取

**问：** 怎么实现比较好？让 LLM 自动判断四种类型的记忆。

**答：** 经过 brainstorming，确定方案：

- **触发**：每轮 AgentLoop 结束后异步执行
- **LLM 调用**：用独立可配置 model（默认 haiku），传已有索引 + 本轮对话
- **输出格式**：通过 tool call（`memory_upsert` / `memory_delete`），代码只负责落盘
- **去重和更新**：LLM 看已有索引判断新增/更新/跳过/删除
- **容错**：失败只记日志，不阻塞主对话

实现涉及 4 个文件：types.ts 新增 tool schema、extractor.ts 重写、chat-service.ts 适配新签名、config/index.ts 新增 memory 配置段。

---
但你应该注意到了，MewCode 到现在为止还是个"被动响应"的工具——所有操作都得用自然语言描述，连「清空当前会话」「看一下当前状态」「列出历史会话」这种高频动作也不例外。下一章我们要给 MewCode 加上 Slash Command 内置命令框架，把 `/clear`、`/status`、`/session list` 这些一键直达的入口接进来，让交互层从"全靠对话"过渡到"对话 + 命令并用"。

### turnRange.start 偏移错误导致用户消息被排除

**问：** 说了"记住：以后都用中文回复"，为什么没有生成记忆文件？

**答：** 两个 bug 叠加导致：

**Bug 1 — turnRange 不含用户消息：** `scheduleMemoryExtraction` 中 `prevCount` 取的是用户消息 push 后的长度（N+1），而 `turnRange.start = prevCount`，导致提取范围从 assistant 回复开始，用户的偏好指令没传进去。修复：`start: Math.max(0, prevCount - 1)`。

**Bug 2 — DeepSeek 不支持 tool call：** 提取用 `memory_upsert` / `memory_delete` 的 tool schema 传给 LLM，但 DeepSeek 的 Anthropic 兼容端点忽略 tools 字段直接返回纯文本，`collectToolCalls` 收不到 `tool_use` chunk，静默返回空结果（无错误、无日志）。

修复：改为文本 JSON 输出。LLM 在回复末尾输出 ` ```json [...]``` ` 代码块，代码层用 `extractJsonFromText()` 解析 JSON 并执行对应操作。不依赖 tool call，兼容所有 LLM API。

**Bug 3 — .gitignore 过宽：** `memory/` 匹配任意层级，连 `src/memory/` 都被忽略。修复：改为 `/memory/` 仅匹配根目录。

### 测试也是按照这个方式验证的

**问：** 改了提取方式后测试怎么适配？

**答：** mock provider 从产出 `tool_use` chunk 改为产出 `text` chunk（包含 ` ```json ``` ` 代码块）。4 个测试用例保持不变：upsert 写入、空数组跳过、delete 删除、错误传播。
