# 流式输出与 Chunk

## 一句话理解

大模型从概念上按 token 逐步生成内容，API 服务端把生成增量包装成 SSE 事件发送；Codia 再把不同厂商的事件转换为统一的 `Chunk`，一边实时显示，一边累积完整回答和工具调用。

## 核心概念

### Token、Chunk 不是一回事

- **Token**：模型处理和生成文本的基本单位，可能是一个字、一个词或词的一部分。
- **API 增量事件**：厂商把一个或多个生成增量包装成一条流式消息。
- **网络数据块**：HTTP 底层某次读取到的字节，切割位置任意，甚至可能把一条 JSON 从中间切开。
- **SSE 事件**：以空行分隔的一条完整 `data: ...` 消息。
- **Codia `Chunk`**：Codia 内部统一的一条 Provider 流事件，不一定是文本，也可能是思考、工具调用、用量、错误或结束信号。

因此它们通常不是一一对应的：

```text
模型逐步生成 token
        ↓ API 服务端组织增量
一个或多个 token 组成 SSE 事件
        ↓ 网络传输可能任意切割
一个 SSE 事件可能分散在多个网络数据块中
        ↓ Codia 解帧并转换
Codia Chunk（统一事件）
```

`Chunk` 更适合理解成“流水线向下游传递的一条事件”，而不是固定大小的一段文本。

### `Chunk` 的 TypeScript 语法

`Chunk` 是可辨识联合类型：

```ts
type Chunk =
  | { type: "text"; content: string }
  | { type: "thinking"; content: string }
  | { type: "done" }
  | { type: "tool_use"; call: ToolCall };
```

竖线 `|` 表示“或者”。所有成员都有 `type` 字段，TypeScript 可以根据它自动收窄类型：

```ts
switch (chunk.type) {
  case "text":
    console.log(chunk.content); // 这里确定存在 content
    break;
  case "tool_use":
    console.log(chunk.call.name); // 这里确定存在 call
    break;
}
```

第一项前也写 `|` 只是统一排版，不改变语义。

## 完整工作流程

### 1. 请求厂商的流式 API

`LLMProvider.streamChat()` 统一返回 `AsyncIterable<Chunk>`。Anthropic 和 OpenAI 分别构建自己的请求体，但 Agent Loop 只依赖这个统一接口。

```text
AgentLoop
  → provider.streamChat()
  → fetch(..., stream: true)
  → response.body
```

### 2. 从网络数据中还原 SSE 事件

网络读取边界不等于 SSE 事件边界。例如一条事件可能被读成两段：

```text
第 1 次读取：data: {"type":"content_
第 2 次读取：block_delta", ...}\n\n
```

`parseSSEStream()` 使用 `buffer` 累积字符串，再按 `\n\n` 分割完整事件；最后一个未完成片段继续留在 `buffer` 中等待下一次读取。

### 3. 把厂商事件映射成统一事件

`mapToChunk()` 同时识别两类协议：

| 厂商事件 | Codia 事件 |
|---|---|
| Anthropic `text_delta` | `text` |
| Anthropic `thinking_delta` | `thinking` |
| Anthropic `input_json_delta` | `tool_input_delta` |
| OpenAI `delta.content` | `text` |
| OpenAI `delta.tool_calls` | `openai_tool_delta` |
| 用量事件 | `usage` |
| 结束事件或 `[DONE]` | `done` |
| 错误 | `error` |

这层屏蔽了文本、用量和错误等常见差异。工具调用还需要在各自 Provider 中进一步组装。

### 4. 边转发，边累积

`StreamCollector` 同时承担两种需求：

1. 每收到一个事件就立刻 `yield`，让 TUI 低延迟显示。
2. 在内部累积 `fullText`、`toolCalls`、`usage` 和错误状态，让 Agent Loop 在本轮结束后作判断。

```text
Provider Chunk
     ├── 立即 yield → TUI
     └── accumulate → 本轮完整结果
```

### 5. 降低终端刷新频率

TUI 收到 `text` 或 `thinking` 后，不会让每个增量都直接触发一次 Ink 重绘。`StreamBuffer` 先在内存中追加内容，默认每 32ms 合并刷新一次 React state，在实时感和渲染成本之间折中。

## 工具调用为什么需要中间事件

文本片段可以收到就显示，但工具参数必须先组成合法 JSON，才能执行。

假设模型想调用：

```ts
read_file({ filePath: "src/app.ts" });
```

参数可能被分批发送：

```text
{"file
Path":"src/
app.ts"}
```

前两个片段都不是完整 JSON，因此 Provider 先保存为中间事件。这里的“中间事件”就是尚未能交给 Agent Loop 执行的半成品通知。

### Anthropic

Anthropic 在 Codia 中对应：

```text
tool_use_start
tool_input_delta × N
        ↓ Provider 累积 inputJson
tool_use（完整 ToolCall）
```

`tool_use_start` 保存工具 ID 和名称，连续的 `tool_input_delta` 追加 JSON 片段。当前实现遇到下一个非工具参数事件、新工具开始或流结束时，提交待处理工具，再执行 `JSON.parse()`。

### OpenAI

OpenAI 在 Codia 中对应：

```text
openai_tool_delta × N
        ↓ 按 index 分组累积
tool_use（完整 ToolCall）
```

OpenAI 可能让多个工具调用交错输出，因此 `openai_tool_delta` 带有 `index`。Provider 使用 `Map<number, PendingTool>` 分别累积每个工具的 `id`、`name` 和 `arguments`，遇到非 `openai_tool_delta` 事件或流结束时统一提交。

## 怎么知道参数拼接完了

Codia 不通过数右花括号，也不通过每次尝试 `JSON.parse()` 来猜测完成状态，而是利用厂商流中的事件边界：

```text
连续参数增量
      ↓
出现下一类事件或流结束
      ↓
认为当前参数块结束
      ↓
JSON.parse 完整字符串
      ↓
生成 tool_use
```

不能简单地把“当前恰好是合法 JSON”当作“厂商已经发完”，因为内容完整性和协议完成状态是两回事。可靠的结束依据应该来自协议边界。

## 为什么要支持不同厂商

Anthropic 和 OpenAI 的请求、响应及工具调用格式不同：

- Anthropic 使用 `/v1/messages`，工具参数通过 `content_block_start` 和 `input_json_delta` 逐步给出。
- OpenAI 使用 `/v1/chat/completions`，工具参数位于 `choices[0].delta.tool_calls[]`，并用 `index` 区分并行调用。

Provider 层负责翻译这些差异，对 Agent Loop 输出统一的 `Chunk`。因此上层调度和 TUI 不必为每家厂商重复实现。

## 易错点与当前限制

### 1. 不要混淆三层“块”

网络数据块、SSE 事件和 Codia `Chunk` 是三种不同粒度。只有 Codia `Chunk` 是应用内部定义的业务事件。

### 2. `openai_tool_delta` 不是最终工具调用

它只是 OpenAI 工具调用的参数碎片。只有 Provider 拼接完成后产生的 `tool_use` 才能进入 Agent 调度。

### 3. Anthropic 也有中间事件

`tool_use_start` 和 `tool_input_delta` 实际承担 Anthropic 专属中间事件的角色，只是名称没有 `anthropic_` 前缀，因此当前命名不对称。

### 4. 公共 `Chunk` 混入了厂商内部事件

当前 `Chunk` 同时包含最终公共事件和三种解析中间事件：

```text
tool_use_start
tool_input_delta
openai_tool_delta
```

这是为了复用解析管线的简化设计。更清晰的方案可以把它们拆进 Provider 内部的 `RawProviderEvent`，公共 `Chunk` 只保留完整的 `tool_use` 等厂商无关事件。这是改进建议，不是当前实现。

### 5. 完成边界目前是间接判断

当前代码主要通过“出现非参数事件”或“流结束”提交工具调用，没有把 Anthropic 的 `content_block_stop` 显式建模为完成事件。显式引入 `tool_input_end` 会更严谨。这也是改进建议。

## 小结

- 模型概念上逐 token 生成；chunk 是 API 或程序对增量数据的包装。
- Codia `Chunk` 是统一 Provider 事件，不是固定大小文本，也不等于网络读取块。
- SSE 解析必须用 buffer 处理任意网络切割，再按空行恢复完整事件。
- 文本可以即时显示；工具参数要等 JSON 片段拼接完成后才能生成 `tool_use`。
- Provider 层吸收 Anthropic/OpenAI 协议差异，Agent Loop 和 TUI 只处理统一事件。
