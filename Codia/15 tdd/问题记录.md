## 2026-07-04

### 开发 Codia 的 /tdd Skill 应该选用哪个 Skill

**问：** 想给 Codia 增加一个 `/tdd` 测试驱动开发功能，但需求不明确，已安装很多 skill，用哪个更合适？

**答：** 需求不明 + 新功能 → 先用 `superpowers:brainstorming` 澄清需求和设计；再选 `codia-spec` 或 OpenSpec（`openspec-propose`/`openspec-explore`）落地 spec/plan/task；开发阶段可用 `superpowers:test-driven-development` 或 `tdd` skill 指导实现。

---

### /tdd 与现有 /test 的区别

**问：** `/tdd` 应该覆盖 TDD 的哪个范围？

**答：** `/test` 负责“运行已有测试并分析失败原因”。`/tdd` 应是编排层，自动循环推进 `red → green → refactor`：理解需求 → 确定 seams → 写失败测试 → 最小实现 → 重构，直到行为被测试覆盖。

---

### /tdd 的设计决策

**问：** `/tdd` 具体形态和实现方式怎么定？

**答：** 最终决策：
- 单文件内置 Skill：`src/skill/builtin/tdd.md`，与 `/commit`、`/review`、`/test` 形态一致；
- `mode: inline`，因为要在当前会话读写文件；
- `allowedTools: [Read, Write, Edit, Bash, LoadSkill]`；
- 全自动推进循环，无需每轮确认；
- `/tdd` 内部通过 `LoadSkill("test")` 复用 `/test` 的测试执行与失败分析能力。

---

### 为什么只新增一个 Markdown 文件就能实现 /tdd

**问：** 就只新增了一个 md 文档就实现了 tdd 的能力吗？

**答：** 是的。Codia 的 Skill 是“提示词框架”：Skill 文件 = YAML frontmatter（元信息）+ Markdown body（SOP）。Skill 系统（loader/registry/activator/command）已建好，`/tdd` 只需定义好 SOP，Agent 激活后按指令行动。测试运行、文件编辑等动作能力已由 `Bash`、`Read`、`Write`、`Edit`、`LoadSkill` 等工具提供。

---

### /tdd 的实现思路

**问：** 给我说一下 `/tdd` 的实现思路。

**答：** 分三层：
1. **基础设施**：Skill 系统负责扫描、注册、激活；
2. **Skill 定义**：`tdd.md` 的 frontmatter 声明模式与工具白名单，body 写 red-green-refactor 循环 SOP；
3. **运行时**：用户输入 `/tdd <需求>` 后，Agent 按 SOP 维护行为清单，逐轮写失败测试、调 `/test`、写最小实现、重构，直到清单完成并输出报告。
