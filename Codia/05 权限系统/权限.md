怎么在不过度限制 Agent 能力的前提下，确保它不做危险的事
三种威胁：
- prompt 注入：在 prompt 中注入了恶意脚本
- 越权操作：只想改 bug ，但是却进行了代码重构 
- 数据泄露：就算是只读，读到了隐私配置，又进行重复，那么也可能造成信息泄露

第二道防线
		路径沙箱：确保所有文件操作（ReadFile、WriteFile、EditFile）只在允许的目录范围内进行。
		核心逻辑三步走
```
		function validatePath(requestedPath, allowedRoots):
		    // 1. 转为绝对路径
		    absPath = toAbsolutePath(requestedPath)
		
		    // 2. 解析符号链接（防止 symlink 逃逸）
		    realPath = resolveSymlinks(absPath)
		
		    // 3. 前缀检查：realPath 必须在 allowedRoots 之内
		    for root in allowedRoots:
		        if realPath.startsWith(root):
		            return OK
		    return error("路径超出沙箱范围")```
		
			每一步都有原因：
			
			Step 1 — 转绝对路径：模型可能传相对路径 ../../etc/passwd。不转成绝对路径，前缀匹配就是空谈。
			
			Step 2 — 解析符号链接：这是最容易漏的绕过手段。攻击者在项目目录内建个软链接 link.txt → /etc/passwd。如果只看字面路径，/project/link.txt 确实在沙箱内，检查通过。但 resolveSymlinks 之后得到 /etc/passwd，立刻暴露。
			
			文档还处理了一个细节：WriteFile 创建新文件时文件还不存在，symlink 解析会失败。此时退而求其次，解析父目录的真实路径——父目录在沙箱内，新文件就没问题。
			
			Step 3 — 前缀匹配：realPath 必须以 allowedRoots 中某个目录开头。
		
```
第三道防线 权限规则
用规则语法来限制工具哪些可以执行，哪些不可以执行。
```
语法：ToolName(content_pattern)

工具名 + 括号里的内容匹配模式，支持 glob：

# 允许所有 git 命令
- rule: Bash(git *)
  effect: allow

# 禁止 force push
- rule: Bash(git push --force*)
  effect: deny

# 允许读取 src 目录
- rule: ReadFile(/project/src/*)
  effect: allow

# 禁止读取环境变量文件
- rule: ReadFile(*.env*)
  effect: deny
```

权限文件为 permission.yml，分为三个等级（三层）：本地、项目、用户。用户点击允许了某些命令之后，自动生成对应的allow 规则放到 permission 里。
```
~/.mewcode/permissions.yaml              ← 用户全局偏好（最低）
{project}/.mewcode/permissions.yaml      ← 项目级，团队共享
{project}/.mewcode/permissions.local.yaml ← 本地覆盖（最高，不提交 git）
```
安全原则：三层里只要有任何一个拒绝了，其他层的 allow 都覆盖不掉

**第四道防线**：**权限模式**
四种权限模式
- default 放行只读，编辑和 Bash 需要用户确认
- acceptsEdit 放行只读和编辑，Bash 需要确认
- plan 安全性等同于 default，主要靠 system prompt 进行限制
- bypassPermissions 模式 依旧拦截‘rm -f’等危险命令，编辑、Bash 命令放行

**第五道防线** HITL human in the loop 人在回路，需要人手动确认
**三个选项**：是/否/始终允许，「始终允许」会创建一条 allow 放到权限配置文件中，这样就完成了一个权限学习循环
**HITL 实现机制：** Agent 遇到一个权限请求事件，就会阻塞来等待用户的回复。UI 层接收到事件来渲染对话框，等用户选择后再把决策通过同步原语回传给 Agent Loop，Agent 收到之后继续执行。

**五层权限层的决策链**
如果上一层能决策就直接返回，不能决策才往下走

在 Agent Loop 中，拒绝不等于终止，被拒绝的工具调用会产生一个isError：true，这个结果通过正常的消息流拼接进入到对话历史，模型在下一轮循环中会看到这个错误，然后继续执行。

虽然五层都很简单，但是加一块就是一个完善的权限系统
