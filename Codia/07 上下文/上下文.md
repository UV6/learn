# 理论学习：上下文压缩与 Token 管理

> **架构定位**：本章实现的是记忆层的上下文管理组件，让 Agent 能在有限的 Token 窗口内长时间工作。

---

## 一、问题：Agent 为什么越跑越慢？

用 AI 编程助手（如 MewCode/Claude Code）跑复杂任务时，你会发现一个现象：**越到后面，响应越慢，Token 用量越来越高**。第一轮对话可能只花 1000 多 input tokens，到第二十轮就飙到十几万。

**根本原因**：Claude API **是无状态的**。每一轮请求都必须把完整对话历史重新发送。Agent 读过的每个文件、执行的每条命令、产生的每段输出，全部堆积在消息列表里，Token 数随对话轮次**线性增长**。

而上下文窗口有上限（如 200K tokens）。1000 行源文件约 15,000 tokens，读 10 个就是 15 万，加上命令输出、搜索结果、AI 回复，很容易逼近上限。一旦撞墙，API 直接报错，Agent 就瘫了。

> **核心问题**：在有限的 Token 预算内，保留最有价值的信息，丢掉可以牺牲的信息。

---

## 二、Token 都花在哪了？

直觉以为 Token 主要花在用户问题和 AI 回复上，实际上：

| 消息类型 | 典型 Token 量 | 占比 |
|---|---|---|
| 固定开销（角色设定、工具描述等） | 5,000–15,000 | ~10% |
| 用户输入 | 50–500 | ~1% |
| AI 文字回复 | 200–2,000 | ~3% |
| 工具调用请求 | 50–200 | ~1% |
| **工具结果（文件内容）** | **5,000–20,000** | **~55%** |
| **工具结果（命令输出）** | **500–5,000** | **~20%** |
| **工具结果（搜索结果）** | **500–3,000** | **~10%** |

**工具结果占了 Token 消耗的 85%**，且很多"保质期"很短。比如第 3 轮读了 `handler.py`，第 5 轮改了它，到第 10 轮那个旧版本还占着预算但已经过时了。

> **结论**：压缩应该从工具结果下手，用户的对话反而是需要着重保留的核心。

---

## 三、两层压缩策略：能轻则轻

整体设计哲学四个字：**能轻则轻**——轻量手段能扛住的，绝不动用昂贵的全量摘要。

| 层级 | 手段 | 信息损失 | API 开销 | 触发条件 |
|---|---|---|---|---|
| 第 1 层 | 大结果存磁盘 | 几乎为零 | 零 | 工具结果超阈值 |
| 第 2 层 | 摘要旧消息 + 保留近期原文 | 中 | 高（一次 API 调用） | Token 数逼近窗口上限 |

---

## 四、第 1 层：大结果存磁盘（预防层）

### 4.1 单个工具结果超限

当工具输出超过 **50,000 字符**（约 12,500 tokens），系统不把它完整放入对话历史，而是：

1. **完整内容写入磁盘文件**
2. 对话历史里只放一个**预览 + 文件路径**

模型看到的是：
```
<persisted-output>
输出太大（80KB），完整内容已保存到：
.mewcode/session/tool-results/toolu_abc123.txt

预览（前 2KB）：
=== 测试运行结果 ===
PASS: TestUserCreate (0.02s)
...
</persisted-output>
```

如果模型后续需要完整内容，用 ReadFile 读取即可。这一层**零额外 API 开销、零信息损失**。

### 4.2 每条消息的聚合限制

单个结果不超限，但并行调用 10 个工具各 40K 字符，合计 400K 也会撑爆。所以还有**每消息聚合上限 200,000 字符**，超了就挑最大的存盘，直到降到预算以内。

### 4.3 幂等写入

存盘文件名用 `tool_use_id`，以 `wx` 模式写入（文件已存在则跳过）。因为每轮发 API 前都会重新检查，同一个结果不需要反复写盘。

### 4.4 Prompt Cache 与决策冻结（关键设计）

这里有一个容易被忽视但非常重要的设计约束。

**背景**：Codia 不改原始对话记录，`conversation.history` 始终存工具返回的**原文**。每轮发 API 前，遍历整段对话重新应用替换决策，生成新的消息列表。如果某一轮的替换决策跟上一轮不同，对话前缀就变了。

**前缀一变 → Prompt Cache 失效**。Anthropic 的 Prompt Cache 原理是：如果前缀逐字节一致就跳过处理直接复用。Cache 持续命中，每轮只需处理增量；Cache 反复失效，每轮都要重新处理所有内容，延迟和成本都上去。

**解决方案：决策冻结**

系统维护一个 `ContentReplacementState`，记录每个 `tool_use_id` 的替换决策：

- **一旦决定"不替换"** → 整个会话内永远不变，后续直接跳过
- **一旦决定"替换"** → 每轮用完全相同的预览字符串原样重放

每轮面对工具结果时分三类处理：
```
已替换过的     → 缓存预览字符串原样重放，不重新生成
已决定不替换的  → 永远保留原文，不再评估
新产生的       → 正常评估，做完决策后冻结
```

**为什么冻结比每轮重新评估更好？** 理论上每轮评估同一条消息会得出相同结论，但重新评估意味着每轮都要排序、检查磁盘文件、重新生成 preview。环境有变化（如 spill 文件被误删），结果就可能漂移。冻结从设计上消除了这种可能，确保前缀逐字节稳定，Cache 一路命中。

> **关键认知**：上下文管理不仅要考虑"保留什么信息"，还要考虑"**改动对话前缀的代价**"。很多人自己做 Agent 时会每轮重新评估所有工具结果，导致 Cache 命中率为零，白白为重复处理前缀买单。

**关于这一块自己的理解**
为了防止工具放行改变导致缓存失效可以把稳定的工具列表与变化的是否放行进行解耦，tools 部分的工具列表不变，把变化的部分放到 message 里面，这样前缀不变是不会影响 prompt 命中的

## 五、第 2 层：Auto-Compact（兜底层）

当第 1 层不够用、上下文仍逼近爆炸时，动用最后手段：**调用 LLM 生成对话摘要**，把较早消息换为摘要，保留近期原文。

代价最高（额外 API 调用 + 信息损失），但压缩效果最强。

### 5.1 触发阈值计算

以 200K 窗口为例：
```
上下文窗口          200,000
- 摘要输出预留       -20,000    ← 摘要本身也要占空间
= 有效窗口          180,000
- 安全余量           -13,000    ← 防 Token 估算误差导致临界抖动
= 自动压缩阈值      167,000
```

**为什么用固定值而非百分比？**

Buffer 保护的是"单轮波动"而非"总量"。一次 ReadFile 不管窗口是 200K 还是 1M 都是 ~15K token，所以 buffer 应该是**固定值**。

如果用"90% 触发"：
- 200K 窗口：180K 触发，剩 20K，刚好装一个文件 ✅
- 1M 窗口：900K 触发，剩 **100K**，够装 7 个文件 ❌ 浪费

如果提高到 98%：
- 200K 窗口：196K 触发，剩 **4K**，一次文件读取就撞墙 ❌ 太危险

**不存在一个百分比能同时适配所有窗口大小**。固定值 33K（20K + 13K）在任何窗口下都恰好够"挡住一到两次大工具调用"。

**13K 安全余量的精度保证**：MewCode 并不是字符数粗估，而是锚定上一次真实 API usage（Anthropic 返回的精确值），只在最后新增的一小段用字符数/3.5~4 估算。误差被压缩在极小范围，13K 才够稳。

**20K 摘要预留**：由两头夹出——下限（复杂会话摘要约 15K–18K tokens，低于此有截断风险）和上限（预留越大有效窗口越小，摘要用不掉纯浪费）。

### 5.2 Token 估算方法

不是简单用字符数粗估。MewCode 的做法：
- 历史部分：锚定上一次 Anthropic 返回的精确 usage（input + cache_read + cache_creation + output）
- 增量部分：只用字符数 ÷ 3.5~4 做粗略估算

因为绝大部分历史是精确的，估算误差只发生在最后几轮，所以才可以用 13K 的余量兜住。

### 5.3 摘要 Prompt 设计

质量关键在于**结构化摘要**，明确指定 9 个部分：

1. **主要请求和意图**：用户到底想做什么
2. **关键技术概念**：讨论过的重要技术点
3. **文件和代码段**：涉及哪些文件，关键代码保留
4. **错误和修复**：遇到了什么错、怎么解决
5. **问题解决过程**：解决问题的思路和方法
6. **所有用户消息**：用户原话尽量**原文保留**（不能改写！）
7. **待办任务**：还没完成的事
8. **当前工作**：最近在做什么（最详细）
9. **可能的下一步**：接下来打算做什么

> ⚠️ **第 6 条至关重要**：用户原话包含意图、偏好、语气。如果用户说"不要用 `interface{}`，用 `any`"，摘要成"用户偏好现代语法"，模型可能给出 `type any = interface{}` 这种不符合用户意图的东西。这是优先级指引，空间不够时 LLM 自然会取舍。

### 5.4 两阶段生成

Prompt 要求 LLM 先产出 `<analysis>` 草稿梳理思路，再产出正式 `<summary>`。实验证明这显著提升摘要质量。`<analysis>` 作为中间产物用完即弃，只有 `<summary>` 被保留。

### 5.5 禁止工具调用

摘要 API 调用复用了原对话的消息上下文，工具定义也会一起传入。必须明确禁止模型调用工具（开头声明 + 结尾重申），或者更彻底的做法：调摘要 API 时直接不传 `tools` 参数。

### 5.6 压缩后恢复

不是把整段对话全换摘要，而是"**旧消息摘要 + 近期原文保留**"：
- 从尾部保留约 **1 万 token / 至少 5 条消息**原文
- 不切断 `tool_use ↔ tool_result` 的配对

同时重新注入：
- **最近访问的文件**：最多 5 个，每个最多 5,000 Token
- **技能定义**：如果用过 Skill，重新注入（预算 25,000 Token）
- **工具列表**：重新声明所有可用工具

压缩后的消息结构：
```
[system]    系统提示词...
[user]      【摘要消息】早期对话的摘要：
            用户要求重构 auth 模块...修复了 handler.py 的 bug...
            
            如需完整细节，请读取会话记录：
            /path/to/session.jsonl
            
            【恢复文件】handler.py 当前内容：...
            【恢复文件】middleware.py 当前内容：...
[user]      好，接下来帮我更新 API 文档    ← 保留的近期原文
[assistant] 我来看一下当前的文档结构...
...
```

> 摘要、会话记录路径、恢复文件拼接在**同一条 user 消息**里。保留会话记录路径是为了防止模型对缺失细节进行"脑补"。

### 5.7 熔断与重试

- **连续失败 3 次** → 停止自动触发，交给用户手动处理（防止死循环）
- **摘要请求超限** → 丢弃最旧消息组重试，最多 3 次，还不行就丢 20%
- **紧急压缩**：正常对话 API 返回 `prompt_too_long` 时，Agent Loop 捕获错误立刻触发 `ForceCompact`，压缩后重试原请求。这是最后一道自救机制。

与自动压缩的关系：**自动压缩是预防，紧急压缩是治疗**。

## 六、手动 `/compact`

用户可随时手动触发，两个典型场景：

- **预防性压缩**：接下来要读大量文件，提前腾空间
- **话题切换**：从调试 bug A 切换到 bug B，清理旧上下文噪声

手动压缩的安全余量只有 **3,000**（比自动的 13,000 小），因为用户主动触发意味着他知道自己在做什么。

## 七、两层的协作关系

每轮 API 调用前的执行顺序：

```
准备发送请求
  → 第 1 层：applyToolResultBudget（大结果存盘，管单条大小）
    → 如果逼近 167K
      → 第 2 层：autoCompact（摘要旧消息 + 保留近期原文）
  → 再次 applyToolResultBudget（处理注入的 system reminders）
  → 发送 API 请求
```

两层解决不同维度的问题：
- **第 1 层**：管**单条消息**的大小，防止一次超大工具输出撑爆上下文
- **第 2 层**：管**对话历史**的累积长度，防止几十轮下来 Token 总量超限

典型场景一：Agent 连续读了 20 个小文件，每个都没超单条阈值，第 1 层不触发。但累积总量逼近 167K，第 2 层触发，旧消息被摘要为几千 Token + 近期原文。

典型场景二：Agent 执行了一条命令输出 80K 日志，第 1 层存盘只留预览。总量远不到阈值，第 2 层不触发。

---

## 八、小结

上下文管理是 Agent 从"短任务玩具"变成"**长时间工作伙伴**"的关键能力。

核心设计哲学：**能轻则轻**。

- 先用**存盘**（零损失零成本）挡住大工具结果
- 再用**摘要**（昂贵但强力）兜底长对话累积

真实生产级 Agent 可能还在两层之间加入中间手段（如 Claude Code 的 microcompact：按时间/数量清理过期工具结果），但核心骨架就是这两层。对自建 Agent 来说，这已经足够撑起一个能长时间稳定工作的系统。

> **容易被忽视的关键认知**：上下文管理的好坏，不仅影响 Token 用量，更直接影响 **Prompt Cache 命中率**。设计时绝不能只考虑"保留什么"，还必须考虑"改动前缀的代价"。决策冻结机制正是为了确保前缀稳定性而引入的，这是很多人自己造轮子时容易踩的坑。
