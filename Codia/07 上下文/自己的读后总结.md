### 第一层：大结果落库

一轮对话中，单个结果超过阈值就写入磁盘，历史记录只放一个预览+路径。但是只处理单个文件的阈值不够，当多个文件不超过阈值，但是总量超过阈值的时候，也需要进行落库：处理方法是从大到小存盘，直到总量在预算内。
单个文件阈值：50000 字符（约12500token，英文字符中，token=总字符➗4，因为coding 肯定是英文更多，这样计算。），每条消息的聚合阈值：200000 字符

**幂等写入**
问题背景：一个会话中会有多轮对话，所以每次调用 api 之前都要重新遍历整个会话，判断是否有文件超限，所以在第一轮超限的文件，不做处理的话可能后面每轮都会进行落库。所以用 tool_use_id 来作为文件名+wx 模式（进行文件写入，如果文件存在就失败，不会覆盖，不会重写）
但是这样还是不是很优雅，下面的决策冻结才是解决这个问题的主力，wx 只是兜底

**prompt cache和决策冻结**
「考虑到改动对话前缀的代价」
因为第一层大结果落库每轮对话都要检查所有工具结果是否超限，但是这里有个坑。先陈述一个背景：codia 不会修改对话记录，对话记录里存的永远都是工具返回的原文，但是在 prompt 中进行第一层大结果落库之后，会用【标记】对工具原文进行覆盖，比如说（tool_use_id+落库标记）。
每轮发 API 前，系统都会遍历整段对话，把决策结果替换上去，生成一份新的消息列表发送给大模型。也就是说，结果可能到某一轮就会用标记覆盖，而这个标记会修改，那么这样【对话前缀】就不同了。那么 prompt cache 就失效了。
参考 claude code 的决策：系统维护一个contextReplacementState，记录每个tool_use_id 的替换决策。规则是：
	一旦某个工具结果在某一轮被决定「不替换」，这个决定在整个会话内永远不变，后续轮次直接跳过，不再重新评估。反过来，如果决定了「替换」，那每一轮都用完全一样的预览字符串去替换，不重新生成，确保前缀逐字节不变。
	这里的一轮是指：调用 api+拿到回复+执行工具+准备下一次请求 的流程（循环轮次），而不是一轮对话（一轮对话里会有多轮循环轮次）
每一轮面对工具结果，系统会把他们分成三类：
- 已替换过的 -- 用标记进行覆盖
- 已决定不替换的 -- 保留原文，且不再评估
- 新产生的 -- 评估后判断是否需要替换

### 第二层：上下文压缩
##### 压缩阈值
按照固定值进行压缩而不是按照百分比进行压缩，如果1M 上下文，10% 就是 100K，是 200K 上下文的一半了，这里的固定值也是一个经验值。
比如说 200k 的上下文，预留 20k 给摘要，然后留出 13k 的安全余量，那么超过了 167k 就要触发全量摘要压缩。
这里的13k固定值就是个经验值

##### 摘要prompt 的设计
结构化摘要主要包括九个部分![[Pasted image 20260616173251.png]]

**摘要的两阶段生成**
先生成一个草稿来梳理思路，然后再生成正式的总结，经过实验发现可以显著提高摘要质量。分析阶段先让LLM把对话中发生了什么梳理一遍，然后在此基础上写摘要。

较为复杂的一次会话，加上草稿大概有 15k 到 18k tokens，这里做一些冗余，写到 20k

**禁止工具调用**
在压缩过程中要禁止工具调用，因为生成摘要的时候api 调用复用了原对话，且 LLM 也看得到可用工具，如果不明确禁止，那么 LLM 就可能会去调用工具。
在实践中，在开头和结尾都进行压缩强调，效果会更好：在开头说一次你只能输出纯文本；在结果说：提醒，不要调用任何工具，调用工具会被拒绝，你会失败。
