# Function Calling 与工具系统 — 学习笔记

> 架构定位：工具是 Agent 能力的边界。没有工具的 Agent 只会说不会做。

---

### 没有工具的 Agent = 被绑在椅子上的顾问

你试试跟一个纯 LLM 说「帮我看看项目入口文件里有什么」。它会回答：「抱歉，我无法直接访问你的文件系统。」

这就像雇了一个精通编程的顾问，结果他被绑在椅子上——脑子好使，手脚不能动。你得自己翻文件、粘贴给他看，他看完告诉你怎么改，你还得自己动手改。

Agent 的四个要素是 **LLM + 工具 + 循环 + 反馈**。只有 LLM 没有工具，本质上还是一个聊天机器人。工具系统是让 Agent 从「说」变成「做」的关键一步。

---

### Function Calling 四步协作协议

模型永远只做一件事：**生成文本**。它不能直接操作任何东西。Function Calling 是一个协作协议，让模型"请求"你帮它执行操作。

#### 第一步：告诉模型有哪些工具

调 API 时通过 `tools` 参数告诉模型有哪些工具可用：

```json
{
  "tools": [{
    "name": "ReadFile",
    "description": "读取指定路径的文件内容。返回带行号的文件文本。路径必须是绝对路径。",
    "input_schema": {
      "type": "object",
      "properties": {
        "path": { "type": "string", "description": "文件的绝对路径" }
      },
      "required": ["path"]
    }
  }]
}
```

就像入职第一天 Leader 告诉你公司有哪些系统可以用。

#### 第二步：模型决定调用工具

模型认为需要某个工具时，在回复里输出一个结构化的调用请求：

```json
{
  "role": "assistant",
  "content": [
    {"type": "text", "text": "让我读取这个文件的内容。"},
    {
      "type": "tool_use",
      "id": "toolu_abc123",
      "name": "ReadFile",
      "input": {"path": "/home/user/project/main.py"}
    }
  ]
}
```

关键细节：
- 模型可以在同一条回复里**既输出文本又请求调用工具**，甚至同时请求多个工具
- 每个 `tool_use` 有唯一 `id`，返回结果时要对应
- 这只是一个**请求/建议**，实际执行与否由你的代码决定

#### 第三步：你执行工具，把结果告诉模型

你的代码执行操作，把结果作为 `tool_result` 发回：

```json
{
  "role": "user",
  "content": [{
    "type": "tool_result",
    "tool_use_id": "toolu_abc123",
    "content": "1\tdef main():\n2\t    print('hello')\n3\t"
  }]
}
```

`tool_result` 以 **user 角色**发送——从对话协议角度，工具执行是用户侧做的事，结果是你反馈给模型的。模型只负责思考和决策，不负责执行。

#### 第四步：模型继续

模型收到结果后：
- 信息够了 → 直接给出最终回复
- 还需要更多 → 再请求另一个工具

这种"调工具 → 看结果 → 再调工具"的链条，就是 Agent Loop 的雏形。

#### 一句话总结本质

> **模型负责决策，你负责执行，结果反馈回模型。** 模型永远不直接操作你的系统——这是一个安全边界，也是一个架构边界。

### 工具描述是最值得打磨的部分

**反直觉的事实：工具描述比工具代码更重要。**

模型决定是否用某个工具时，主要依据就是描述。描述不清楚 → 不知道什么时候该用 → 用错工具 → 传错参数。

一个好描述应该包含：

| 信息 | 说明 | 示例 |
|------|------|------|
| 做什么 | 核心功能 | "读取指定路径的文件内容" |
| 什么时候该用 | 典型场景 | "需要查看文件完整内容时" |
| 什么时候不该用 | 反模式提示 | "二进制文件不可用，改 Bash" |
| 参数约束 | 输入限制 | "路径必须是绝对路径" |
| 返回格式 | 输出长什么样 | "返回带行号前缀的文本" |
| 与其他工具的配合 | 工作流建议 | "大文件先 Grep 定位再 ReadFile" |

对比：

```
❌ 差："读取文件"
✅ 好："读取指定路径的文件内容。返回带行号的文件文本。
       大文件建议先用 Grep 定位再 ReadFile 读取指定范围。
       路径必须是绝对路径。二进制文件改用 Bash。"
```

### 工具接口设计

最直觉的设计：名字 + 执行方法。但不够——权限系统需要知道"只读还是破坏性"，参数校验要提前拒绝而非执行到一半才报错，API 需要的工具定义也从这里出。

完整的工具接口：

```
name()              → 工具名
description()       → 工具描述
inputSchema()       → JSON Schema
execute(ctx, input) → ToolResult
isReadOnly()        → 只读？
isDestructive()     → 破坏性？
isConcurrencySafe() → 能否并发？（默认 false，保守优先）
category()          → 分类（file/shell/search）
validateInput()     → 参数校验，失败提前拒绝
```

#### 执行结果：错误也是有价值的信息

```
ToolResult {
    content: string     // 返回给模型的文本
    isError: boolean    // 是否为错误结果
    metadata: map       // 给 UI 用的额外信息，不发给模型
}
```

关键设计：工具执行失败（如文件不存在）包装成 `isError: true` 的 ToolResult 返回给模型——模型看到后会调整策略。**只有真正的系统级错误（内存不足、崩溃）才作为程序级 error。**

#### 通用基础实现：别让每个工具重复造轮子

用 `BaseTool` 做通用实现，每个工具只需写一个工厂函数填入元信息和执行函数：

```
BaseTool { name, description, inputSchema, readOnly, destructive,
           concurrencySafe, category, executeFn, validateFn }

// 具体工具只要写工厂函数：
function newReadFileTool():
    return BaseTool {
        name: "ReadFile",
        readOnly: true,
        concurrencySafe: true,    // 只读操作，可并发
        executeFn: executeReadFile
    }
```

`concurrencySafe` 规则：天然只读的工具（ReadFile、Grep、Glob）返回 true，其他一律 false。Bash 理论上可以按命令判断，但解析 shell 命令不可靠，保守起见不并发。

#### 不同语言的惯用模式

| 语言 | 接口表达 | 基础实现复用 | Schema 处理 |
|------|---------|-------------|------------|
| Go | interface + baseTool struct | 组合（嵌入 struct） | json.RawMessage |
| Python | Protocol / ABC + dataclass | 继承或 mixin | dict / Pydantic |
| TypeScript | interface + 工厂函数 | 闭包或 class | 纯 JSON 对象 |
| Java | interface + 抽象类 | 继承或组合 | Map<String, Object> |

### 工具注册中心

工具注册中心（Registry）是工具的**统一管理入口**，把创建和使用解耦。核心能力：

```
register(tool)        → 注册工具
disable(name)         → 禁用工具
getEnabled()          → 获取所有启用的工具
toAPIFormat()         → 每次调 API 前，把所有启用工具的名称、描述、Schema 组装成 API 格式
```

支持条件启用：

```
// 基础读取始终启用
registry.register(newReadFileTool())
// 写操作需要显式开启
if config.allowWrite:
    registry.register(newWriteFileTool())
// Bash 最危险，单独授权
if config.allowBash:
    registry.register(newBashTool(config.bashTimeout))
```

### 六个核心工具

覆盖 Coding Agent 最基本的操作需求：读代码、写代码、改代码、跑命令、找文件、搜内容。

#### ReadFile — 最简单，但有三个暗坑

1. **行号**：返回内容必须带行号前缀（`1\tdef main():`），否则模型引用代码位置时不精确
2. **大文件**：支持 `offset` + `limit` 分段读取，避免一次消耗大量 token
3. **二进制文件**：读前 512 字节检测 NUL 字符（`\x00`），有则拒绝并提示用命令行工具

错误处理：文件不存在 / 路径非绝对 / 二进制 / 权限不足 → 返回 `isError: true`，提示为什么失败和怎么调整。元信息：只读。

#### WriteFile — 创建和覆盖

完整覆盖写入，不是追加。执行前递归创建父目录（755），文件用 644 权限。返回"成功写入 N 字节"。元信息：非只读，非破坏性。

#### EditFile — 最复杂，也最省 token

一个 500 行的文件只改一行，WriteFile 需要输出全部 500 行浪费几千 token。EditFile 只给 `old_string` 和 `new_string`。

**核心约束：old_string 必须在文件中唯一匹配。**

```
找到 1 次 → 执行替换
找到 N 次 → 报错，提示"请提供更多上下文使其唯一"
找到 0 次 → 报错，模型可能记错了文件内容
```

`new_string` 为空 = 删除。替换成功后返回修改位置附近的几行（带行号）供模型确认。元信息：非只读，非破坏性。

#### Bash — 最强大，也最危险

通过系统 shell 执行任意命令。默认超时 120 秒。stdout + stderr 合并，输出过长截断。

**关键设计：命令退出码的语义解读。** 默认非零 = `isError: true`，但有例外：

| 命令 | 退出码 1 | 含义 |
|------|---------|------|
| grep | 1 | 没找到匹配，正常结果 |
| diff | 1 | 文件有差异，正常结果 |
| find | 1 | 部分目录不可访问 |

这类命令只有退出码 2+ 才算 error。元信息：**破坏性**（唯一），权限系统会触发最严格的确认流程。

#### Glob — 找文件

核心参数：glob 模式（如 `**/*.py`）+ 可选搜索根目录。支持 `**` 递归。自动排除 `.git`、`node_modules` 等目录。结果按修改时间倒序，最多 200 个。

#### Grep — 找内容

核心参数：正则表达式。可选搜索路径、按 glob 过滤文件名、上下文行数。自动排除二进制和 `.git` 等。最多 100 个匹配。输出格式：`文件路径:行号: 匹配行内容`。

#### 六个工具全景

| 工具 | 分类 | 只读 | 破坏性 | 典型场景 |
|------|------|------|--------|---------|
| ReadFile | file | 是 | 否 | 查看文件内容 |
| WriteFile | file | 否 | 否 | 创建/覆盖文件 |
| EditFile | file | 否 | 否 | 精确修改，省 token |
| Bash | shell | 否 | **是** | 编译/测试/执行命令 |
| Glob | search | 是 | 否 | 了解项目结构 |
| Grep | search | 是 | 否 | 搜索函数定义/变量引用 |

典型协作链：**Grep 搜关键词 → 定位文件 → ReadFile 读取 → EditFile 修改 → Bash 编译测试**

### 集成到 LLM 客户端：流式 tool_use 解析

#### 请求侧

每次调 API 时从 Registry 拿工具定义放进请求参数。

#### 响应侧

新增 `tool_use` 和 `tool_result` 两种内容类型，以及对应的流式事件类型。

#### 流式 tool_use 解析（本章最 tricky 的部分）

`tool_use` 的输入参数在流式响应中是**一段一段到的 JSON 碎片**，需要拼起来：

```
content_block_start  → type: "tool_use", id: "toolu_xxx", name: "ReadFile"
content_block_delta  → type: "input_json_delta", partial_json: "{"
content_block_delta  → type: "input_json_delta", partial_json: "\"path\""
content_block_delta  → type: "input_json_delta", partial_json: ": \"/main.py\"}"
content_block_stop   → 块结束，解析完整 JSON → 发送 ToolUse 事件
```

处理三步：`start` 时记下 id/name 并初始化 buffer → 每个 `delta` 追加 `partial_json` 到 buffer → `stop` 时解析完整 JSON。JSON 拼接出错不能崩溃，优雅降级。

如果是普通文本，解析之后直接直接拼接，遇到 SSE 的\n\n 就停止。但是工具的话，工具里面调用工具的信息是用 json 表示的，会有{ },如果单独的一个不完整的 json，解析是会报错的，所以这里存在两次解析，不能像文本信息一样边到边用，需要每个增量追加时遇到结束标志时，才进行解析和执行。
Codia 是这么做的：遇到tool_use_start，就创建一个缓冲区，然后把后面的 tool_use_delta不断追加到缓冲区。如果下一个事件不再是tool_use_start （因为可能调用多个工具）或 tool_use_delta，就认为结束了

### 消息管道的变化

引入工具后，对话不再是简单的 user ↔ assistant 交替：

```
user:      "帮我读入口文件"
assistant: [text: "好的"] [tool_use: ReadFile(path="/main.py")]
user:      [tool_result: "1\tdef main():\n..."]
assistant: "这个文件包含了..."
```

要点：
- `tool_result` 以 **user 角色**发送，不打破 user/assistant 交替格式
- 一条 assistant 消息可同时含 text 和 tool_use，**必须放在同一条消息里**
- 多个 tool_use 和多个 tool_result 通过 **id 配对**

---

### 本章核心三点

1. **决策与执行分离**：模型只管想，代码负责做，结果反馈回来驱动下一步
2. **工具接口不只是"名字 + 执行"**：只读/破坏性标记、输入校验等元信息是权限系统自动化运转的基础
3. **工具描述是回报率最高的投入**：描述写得好，模型选对工具、传对参数的概率显著提升
