工具描述是整个系统中最重要的部分，如果描述不清楚，模型要么不知道什么时候用，或者用错。
一个工具的描述应该包含这些信息：
1. 做什么
2. 什么时候用
3. 什么时候不该用
4. 参数约束
5. 返回格式
6. 在工作流的位置
另外，除了描述，工具还要把权限和执行封装在一起。一个工具接口应该包括这些方法：![[Pasted image 20260610000122.png]]
名称、描述、输入、是否只读、是否具有破坏性、是否并发安全、工具分类
是否有破坏性和是否并发安全会在权限管理中触发最严格的用户确认流程

⭐️**六个核心工具，每个工具都要仔细考量**
ReadFile
- 因为大模型可能会描述某一行有错误，所以要再读文件的前面加一个行号
- 如果一个文件很大的话，一次性读取就太消耗token了，所以需要指定从哪里开始，多少行的一个功能
- 如果一个文件是二进制形式的，那么还需要调用Bash工具进行读取，不能直接读取
- 元信息：只读、非破坏性、file 分类
WriteFile
- 完整覆盖，不是追加
- 得考虑到如果父目录不存在的情况，所以要先父目录，再创建文件
- 写入的确认信息是：确认写入 N 字节到path
- 元数据：非只读、非破坏性、file 分类
EditFile
- 适用场景：比如说有一个500行的文件，而我只需要修改第412行，那么用覆盖的话就不合适
- 匹配过程（唯一匹配才进行修改）：先进行唯一匹配，如果找到没有找到或者找到多个值的话，就让大模型给出更长的上下文（被替换的）再进行判断，直到被替换的在文件中只存在一份！
- 替换成功后再返回带有行号的附近几行内容，让大模型确认是否修改成功
- 如果新字符串为空，则表示删除
- 元数据：非只读、非破坏性、file 分类
Bash
- 用来执行任何 shell 命令，所以对它的权限也最严格。
- 在执行方面，用当前项目根目录作为工作目录，防止影响项目外其他目录。默认超时30 秒来防止死循环
- 在输出方面，把 stdout 标准输出和stderr 错误输出合并到同一个流里，并限制对超过 10000 的字符进行截断，只保留前 2000 和后 8000（因为错误信息更多的在后面）
- 返回格式：用标签包裹 输出内容 和 exit code（默认 非零退出码为出现报错）
	- 命令退出码设计：因为有些命令退出码不一致，有的是 1，有的是 2，所以这里维护一个命令语义表，为不同命令维护不同的退出码。
- 元信息：非只读，破坏性（唯一一个），分类为 shell
Glob
- 找文件名，用匹配模式查找
- 可自动排除.git，.idea 等目录
- 按照修改时间倒序排列，最多返回两百个结果
- 元数据：只读、非破坏性、search 分类
Grep
- 找文件内容的，正则表达式匹配，
- 逐文件逐行匹配，自动排除二进制文件和.git 目录，最多返回 100 个结果
- 输出格式：文件路径：行号：匹配内容
- 元数据：只读、非破坏性、search 分类

集成到LLM 客户端，处理流式 tool_use
- 请求侧：每次 API 调用都把工具列表带上
- 响应侧：增加 tool_use 和tool_result。增加了工具之后，也要增加 context 的类型：
	- tool_use：id、name、input
	- tool_result：tool_use_id（绑定哪个工具的执行结果）、context（工具内容）、is_error（是否为错误执行结果）。

	- 消息管道的变化
		- 从user/assistant 交替加上了 tool_use【assistant 发的，想要调用工具】 和 tool_result【user 发的，返回执行结果给 assistant】，但 user/assistant 交替的骨架没变
		- 一条 assistant 消息包含 text 和 tool_use 两个内容块
		- 如果调用多个工具，那么所有的tool_use块在同一条 assistant消息里，所有的tool_result 在同一个 user 消息里。
- **难点：tool_use 解析**
	- 在流式响应中，文本是一段一段追加的，但是一个工具内容分为若干个json 碎片，比如说 ReadFile 工具： {"path": "/main.py"} ，这个 json 对象要分成判断来发送：
	- 收到context_block_start，记下 id 和 name，初始化一个缓冲区
	- 如果遇到 context_block_delta标签，就把内容追加到缓冲区
	- 如果遇到 context_block_end 标签，缓冲区里的字符就是完整的 json，用它发出调用工具命令
