# 我的初步想法
这一步的目标是：把原来三行的全局指令展开成结构化的系统提示，让稳定的指令走缓存通道省钱省时间，把环境信息和动态指令通过系统级补充消息注入，让 Codia 从「能干活」变成「干得好」。

技术要求：

- 把全局指令按职责拆成若干模块（身份、行为、工具使用、代码规范、安全边界、任务模式、输出风格），按优先级拼装，方便以后插新模块
- 区分稳定内容和变化内容：稳定的指令和工具描述走可缓存通道，环境信息和对话历史走消息通道
- 在工具描述和全局指令里双重强化关键规则，提高模型对「优先用专用工具」「编辑前必先读」这类约定的遵守率
- 用一种带特殊标签的消息形式在运行中注入补充指令，既不污染缓存，也不会被模型当成用户输入来回复
- 会话级开关（如规划模式）的指令按轮次注入，首轮完整、间隔轮次重复、其余精简，控制注入频率
- 通过解析 API 返回的缓存命中字段验证策略是否真生效，再准备一组典型场景做人工对比当定性评估

这一步先不做项目指令文件加载、自动记忆、真实 MCP 接入、自动化评估，这些是后面几章的事。

系统提示结构：

- 七个固定模块按优先级排：身份、系统约束、任务模式、动作执行、工具使用、语气风格、文本输出，最后接环境信息
- 另有几个可选模块排在后面：自定义指令、已激活的 Skill、长期记忆
- 模块之间空行分隔，整体按优先级从前往后拼
