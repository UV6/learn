# Slash Command 总结

Slash Command（斜杠命令）是给 Agent 系统加的一条**本地快车道**：凡是以 `/` 开头的输入，优先走命令解析器，本地直接执行，不进入 Agent Loop，因此**响应更快、消耗更少 token、结果也更确定**。

## 为什么需要 Slash Command

文章的核心动机是：

- 有些操作本来就是 UI 或本地层动作，不需要 AI 参与
- 例如：清屏、查看 token、切换 Plan Mode
- 如果这些操作也走 LLM，就会出现“为了省 token 反而先消耗 token”的尴尬情况

所以 Slash Command 的目标是：

- 绕过 LLM
- 毫秒级响应
- 处理确定性的预定义操作

## 一个命令框架要解决的 3 件事

### 1. 注册

每个命令都要有统一定义，类似一张“身份证”，通常包括：

- `name`：命令名
- `aliases`：别名
- `description`：描述
- `usage`：用法
- `type`：命令类型
- `argPrompt`：缺参数时的提示
- `hidden`：是否隐藏
- `handler`：真正执行逻辑

这些信息集中注册到 `registry`，而不是散落在一堆 `if-else` 里。

### 2. 解析

输入形如：

```text
/compact 保留数据库相关内容
```

解析器要拆出两部分：

- 命令名：`compact`
- 参数：`保留数据库相关内容`

基本规则很简单：

- 以 `/` 开头才算命令
- 第一个空格前是命令名
- 后面都是参数
- 命令名统一转小写，避免大小写差异

### 3. 执行

根据解析出来的命令名，去 registry 里找到对应的命令，然后调用它注册时绑定的 `handler`：

```
cmd = registry.find(“compact”)   // 查出注册时存好的 Command
cmd.handler(ctx)                 // 调用注册时存好的 handler
```

所以**注册里的 `handler`，就是这一步执行时要调的那个函数**。注册只负责”存”，执行才负责”调”。

打个比方：注册就像通讯录里存”张三 → 138xxxx”，执行就是真的拨这个号码。

`handler` 的签名是统一的 `handler(ctx)`，只要拿到 `ctx`，命令就能控制 agent、修改对话、更新 UI、读取配置。

`ctx` 里打包的东西包括：

- 参数
- Agent 实例
- 当前对话
- 当前会话
- UIController
- 全局配置

这样每个命令只关心”做什么”，不用自己到处找依赖。

---

**三步串起来看**：注册决定了”谁管这个命令”，解析决定了”用户想干哪个命令”，执行就是”把那个人叫起来干活”。

## 三种命令类型

文章把命令分成 3 类：

### 1. `local`

纯本地命令，不走 Agent Loop。

例子：

- `/help`
- `/status`
- `/session list`

特点：

- 不耗 token
- 执行快
- 一般只是在 UI 中显示结果

### 2. `local-ui`

本地执行，但会改变 UI 状态。

例子：

- `/clear`
- `/plan`
- `/do`

特点：

- 仍然不走 LLM
- 但会影响界面行为、状态栏、模式切换等

### 3. `prompt`

本质上是“命令化的 prompt 模板”。

例子：

- `/review`

特点：

- 会把预设 prompt 转发给 Agent
- 仍然会消耗 token
- 但省去了用户手写复杂提示词的麻烦

## 提升易用性的几个设计

### 1. 别名

给高频命令更短的写法，例如：

- `/compact` → `/c`
- `/help` → `/h` 或 `/?`

这样能显著减少输入成本。

### 2. 参数提示 `argPrompt`

有些命令必须带参数，比如：

```text
/resume <id>
```

如果用户只输入 `/resume`，系统不应该只报错，而应该提示正确用法。

### 3. Tab 补全

输入 `/` 后按 Tab，可列出所有命令；
输入 `/com` 后按 Tab，可自动补全为 `/compact`。

这让命令系统更像成熟 CLI，而不是生硬的字符串匹配。

## 和 UI 事件循环的集成方式

命令系统必须在“消息发送给 Agent 之前”拦截输入：

1. 用户按回车
2. 先判断是不是 `/` 命令
3. 如果是命令，走命令系统
4. 如果不是命令，才发给 Agent

这里的关键点是：

- 只输入 `/` 时，不报错，而是直接展示命令列表
- 未知命令时，提示用户用 `/help`
- 缺参数时，优先给出 `argPrompt`

也就是：**错误信息不只是报错，还要带下一步指引**。

## 文中给出的 10 个核心命令

### 本地命令（local）

- `/help`
- `/compact`
- `/session`
- `/memory`
- `/permission`
- `/status`：LLM 每次响应都会在流末尾带上 usage 信息，逐层透传到 UI state。/status 只是把这个 state 格式化展示出来，没有额外查询。
### 影响 UI 状态的命令（local-ui）

- `/clear`
- `/plan`
- `/do`

### 转发给 Agent 的命令（prompt）

- `/review`

这组命令体现出一个思路：

- 能本地做的事，就不要交给 LLM
- 必须借助 AI 的事，再通过 prompt 型命令做快捷入口

## 文章最后的边界判断

Slash Command 很适合处理：

- 预定义
- 程序化
- 确定性强
- 高频本地操作

但它也有明显边界：

- handler 是硬编码的
- 修改行为通常要改代码、重新编译
- 用户不能自由扩展命令

因此，Slash Command 解决的是“**内置固定命令**”的问题；更灵活、可扩展、可动态贡献命令的能力，要交给下一章的 **Skill 系统**。

## 一句话总结

Slash Command 的本质，不是“让用户少打几个字”，而是把**不需要 AI 的操作从 Agent Loop 里剥离出来**，建立一套统一的命令注册、解析、执行框架，让本地操作更快、更省 token、更稳定，同时为后续 Skill 系统打下扩展基础。
