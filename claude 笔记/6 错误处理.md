### 核心问题

Agent 跑着跑着报错 `Error: 529 overloaded`，直接崩溃了。没有重试，没有换模型，没有减少上下文——一碰就熄火。

生产环境中 API 错误是常态。三种最常见的故障模式：

- **输出被截断**：模型话说一半，`max_tokens` 用完了
- **上下文超限**：s08 的四层压缩全跑过了，还是超
- **临时故障**：网络抖动、429 限流、529 过载

### 解决方案：三种恢复路径 + 指数退避

LLM 调用包裹在 try/except 里，根据错误类型走不同路径。恢复后 `continue` 回到循环开头重新调用 LLM。

---

### 路径 1：输出被截断

模型话说一半，`max_tokens` 用完了，`stop_reason == "max_tokens"`。

**两级策略**：

1. **第一次发生**：直接把 `max_tokens` 从 8K 升级到 64K（8 倍），重试同一请求。此时不追加截断输出到 messages，保持原始请求干净
2. **64K 还是不够**：保存截断输出，注入续写提示让模型接着继续说，最多 3 次

续写提示的关键措辞：**"直接继续，不要道歉，不要回顾你刚才在做什么。如果被截断在句子中间，就从那里继续。"** 因为如果让模型"总结+重新开始"，它会浪费 token 重复已经输出的内容。

**为什么升级只有一次，续写最多 3 次？** 如果 64K 还不够，再升级也没用（API 上限）。续写 3 次后再截断，说明输出远超预期，继续续写不会有实质产出（CC 源码中还有 diminishing returns 检测：连续 3 次 continuation 且每次 token 增量 < 500，停止）。

---

### 路径 2：上下文超限

LLM 返回 `prompt_too_long`——s08 的四层压缩全跑过了，还是超。

触发 **reactive compact**：比 auto compact 更激进，用字节级精度裁剪到 API 可接受的大小。但只会尝试一次——压缩过一次还是超限，再压缩也不会变小，只能退出。

这和 s08 的应急压缩是同一个机制，s11 把它正式纳入错误恢复框架中。

---

### 路径 3：临时故障（429/529）

网络抖动、429 限流、529 过载——这些不是 bug，是分布式系统的常态。

**指数退避 + 随机抖动**：

```
delay = min(500 × 2^attempt, 32000) + random(0~25%)
```

|第几次|基础延迟|加抖动后|
|---|---|---|
|1|0.5s|0.5~0.6s|
|2|1s|1~1.25s|
|3|2s|2~2.5s|
|4|4s|4~5s|
|7+|32s（上限）|32~40s|

最多重试 10 次。如果服务器返回 `Retry-After` header，优先用那个值。

**529 的特殊处理**：连续 3 次 529 过载 → 自动切换到备用模型（如 Opus → Sonnet），同时清除 pending 的工具结果，告知用户换了模型。

**为什么加随机抖动？** 保证大量并发请求不在同一时刻重试——如果所有客户端在第 3 次重试时都是精确的 2 秒，API 会被瞬间打爆。

---

### 三种机制如何协作

```
agent_loop:
  while True:
    try:
      response = with_retry(         ← 路径 3：429/529 指数退避
        lambda: client.messages.create(...)
      )
    except PromptTooLongError:       ← 路径 2：上下文超限
      reactive_compact → retry once

    if stop_reason == "max_tokens":  ← 路径 1：输出截断
      escalate 8K→64K → retry
      or inject continuation prompt

    if stop_reason != "tool_use":
      return                        ← 正常结束
```

**外层 try/except 捕获 API 异常、`with_retry` 处理瞬态错误、`stop_reason` 检查处理截断。三种恢复机制各管各的错误类型，层次分明。**

### real Claude Code 的复杂度

教学版只讲了 3 条恢复路径，CC 实际有十几种 reason/transition：

|reason|含义|
|---|---|
|`max_output_tokens_escalate`|8K→64K 升级|
|`max_output_tokens_recovery`|续写提示（最多 3 次）|
|`reactive_compact_retry`|reactive compact → 重试|
|`prompt_too_long`|上下文过长的各类处理|
|`collapse_drain_retry`|context collapse 先提交暂存|
|`model_error`|模型自身错误重试|
|`image_error`|图片过大/格式错误专门处理|
|`aborted_streaming`|流式中止恢复|
|`stop_hook_blocking`|注入 blocking error → 模型自纠|
|`token_budget_continuation`|token 用量 < 90% 时继续|
|`blocking_limit` / `max_turns`|达到限制|

**流式错误处理**中有个巧妙设计：413、max_tokens、media error 在 streaming 期间被暂扣不展示——SDK 消费者看不到，只有恢复逻辑能看到。等 streaming 结束后再判断是否需要恢复。这样用户不会看到"半个错误"。

**Diminishing Returns 检测**：token budget 的"继续"不是无限的。连续 3 次 continuation 且 token 增量 < 500 时，系统判断"继续也没有实质性产出"，停止 continuation
