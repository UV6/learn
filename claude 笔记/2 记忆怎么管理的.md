为了解决压缩带来的副作用，压缩后细节会退化，为了防止压缩带来的损失，就用一层文件系统来做持久化，不压缩会话。分为四个部分理解：**存储、加载、写入、整理**
## 存储：md+索引
每个记忆是一个 `.md` 文件，存储在 `.memory/` 目录下，带 YAML frontmatter：
- **`name`**：记忆的唯一标识（kebab-case）
- **`description`**：一行摘要，用于索引查找
- **`type`**：四类之一
- **正文**：记忆的完整内容

四类记忆：
|类型|回答什么问题|示例|
|---|---|---|
|**user**|我是谁、我喜欢什么|"用 tab 不用空格"|
|**feedback**|应该怎么做事|"别 mock 数据库"|
|**project**|正在发生什么|"auth 重写是因为合规驱动"|
|**reference**|东西在哪找|"pipeline bug 的 ticket 在 Linear INGEST-421"|

## 加载：放到系统提示词中常驻+每轮会话开始通过加载记忆清单和关键词匹配来选出关键词，最多五条
**路径一：索引常驻 SYSTEM prompt。** 每轮构建 SYSTEM prompt 时，读取 `MEMORY.md`，把所有记忆的名称和描述注入。这个索引可以被 prompt **cache** 缓存，不需要每轮重新发送，开销很低。Agent 始终知道自己"有哪些记忆可用"。

**路径二：相关记忆按需注入到当前对话。** 每轮开始时，`load_memories()` 用两个步骤选出相关记忆：
1. 把记忆清单（name + description）+ 最近对话，发给 LLM 做一次轻量 side-query，选出真正相关的文件名
2. 如果 side-query 失败（网络错误、JSON 解析失败），降级为关键词匹配——在 name 和 description 中匹配对话中出现的关键词

选出的相关记忆（最多 5 条）读取完整内容，临时注入到当前 user turn 中。不修改 SYSTEM prompt，不影响 prompt cache 的前缀结构。

为什么最多 5 条？控制开销，避免记忆本身把上下文打满。【真实行为】

## 写入：每轮对话结束后，让 LLM 提取偏好、约束、实时保存到 memory 中，再重建记忆
用户不会特意说"记住这个"，偏好通常散落在正常对话中："用 tab 比空格好"、"以后都用单引号"。

`extract_memories()` 在每轮对话结束时运行——条件是模型停止了且没有 tool_use（说明一轮对话告一段落，不是中间的工具调用）。流程：

1. 取最近 10 轮对话
2. 列出现有记忆的 name + description（避免重复提取）
3. 把对话 + 已有记忆清单发给 LLM，要求提取新的偏好、约束或项目事实
4. LLM 返回 JSON 数组 `[{name, type, description, body}]`，如果没新东西则返回 `[]`
5. 有结果就逐个写入 `.memory/*.md`，自动重建索引

这里有个重要细节：提取用的是**压缩前的消息快照**（`pre_compress`），不是压缩后的。因为压缩后细节可能已经丢了，但从快照中提取能保留完整信息。

## 整理：当文件数量达到阈值的时候，把所有记忆发给 llm，合并重复的、过时的、矛盾的记忆。
记忆文件会随时间积累，可能出现重复、矛盾、过时的内容。

`consolidate_memories()` 在文件数达到阈值（教学版设为 10）时触发，把所有记忆发给 LLM：合并重复的、删掉过时的、解决矛盾、保持在合理数量范围内。然后用整理后的结果替换所有记忆文件。

真实 Claude Code 中这个过程叫 **Dream**，有四层门控而不是简单的文件数阈值：

1. 距上次合并 ≥ 24 小时
2. 扫描节流，避免频繁扫描文件系统
3. 至少 5 个会话 transcript 被修改过
4. 文件锁，没有其他进程正在合并
