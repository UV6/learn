### 核心问题

前 19 章每章只加一个机制——适合学习，但真实 Agent 不会只带一个机制运行。一个能长期工作的 coding agent 需要同时拥有 19 种能力：工具分发、权限边界、hooks、计划与任务、技能与记忆、压缩与恢复、后台与 cron、团队与自治、worktree 隔离、MCP 外部工具。

### 难点不是堆功能，而是看清楚它们都挂在循环的哪个位置

s20 不发明新机制，而是把前 19 章的组件全部归位到同一个 `while True` 循环中。循环本身的结构和 s01 完全一样：调用模型 → 检查 `tool_use` → 执行工具 → 结果追加回 `messages`。变化的是循环**周围的 harness 变完整了**。

### 所有组件在循环中的位置

```
用户输入
  │
  ├─ UserPromptSubmit hooks          ← 记录、注入、审计
  ├─ cron / background 通知注入        ← 定时任务 + 后台完成通知
  │
  ▼
压缩管线 (LLM 前)
  ├─ L3: tool_result_budget          ← 大结果落盘
  ├─ L1: snip_compact                ← 裁中间消息
  ├─ L2: micro_compact               ← 旧结果占位
  └─ L4: compact_history (条件)      ← LLM 全量摘要
  │
  ▼
组装 system prompt (LLM 前)
  ├─ identity / tools / workspace    ← 始终加载
  ├─ skills catalog                  ← 启动时扫描，目录级
  ├─ memory index (MEMORY.md)        ← 索引常驻
  ├─ relevant memory content         ← 按需注入（最多 5 条）
  └─ MCP server info                 ← 已连接的 server 列表
  │
  ▼
LLM 调用
  ├─ 429/529 → 指数退避重试          ← 临时故障
  ├─ max_tokens → 8K→64K 升级        ← 输出截断
  ├─ prompt_too_long → reactive      ← 上下文超限
  └─ 连续 529 → fallback model       ← 切换备用模型
  │
  ├─ 没有 tool_use? → Stop hooks → 提取记忆 → 整理 → 返回
  │
  ▼
工具执行
  ├─ PreToolUse hooks + permission   ← 拦截危险命令
  ├─ assemble_tool_pool              ← 内置 + MCP 工具合并
  ├─ should_run_background?          ← 慢操作 → daemon 线程
  │   ├─ 是 → start_background_task → 占位 tool_result
  │   └─ 否 → 同步执行 → tool_result
  ├─ compact 工具?                   ← 触发 compact_history
  ├─ connect_mcp?                    ← 连接后重建工具池
  ├─ spawn_teammate?                 ← 启动队友线程
  └─ PostToolUse hooks               ← 日志、告警
  │
  ▼
tool_result 追加到 messages → 下一轮
```

### 工具池：27 个内置工具 + MCP 动态工具

s20 包含所有 s01-s19 引入的工具：bash、read、write、edit、glob（s01-s02）、todo_write（s05）、task（s06）、load_skill（s07）、compact（s08）、create/list/get/claim/complete_task（s12）、schedule/list/cancel_cron（s14）、spawn_teammate/send_message/check_inbox（s15）、request_shutdown/plan/review_plan（s16）、create/remove/keep_worktree（s18）、connect_mcp（s19）。

`assemble_tool_pool()` 每轮组装：内置工具 + 已连接 MCP server 的工具，带 `mcp__server__tool` 前缀。

### 全书终点

从 s01 到 s20，代码表面越来越复杂，但核心始终没变：

```python
while True:
    response = LLM(messages, tools)
    if not has_tool_use(response.content):
        return
    results = execute_tools(response.content)
    messages.append(tool_results)
```

Claude Code 的复杂性不是"另一个 agent 大脑"，而是一个成熟 harness 的复杂性：模型负责判断和行动选择，harness 负责把环境、工具、权限、记忆、团队和外部能力组织好。

**机制很多，循环一个。**
