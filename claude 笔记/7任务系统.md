任务之间的关系、以及任务的持久化
### 核心问题

TodoWrite 是当前任务的执行清单，存在进程内存里。但如果你让 Agent"搭数据库、写 API、加测试"，它列了一张 TodoWrite 清单就开始写 API。写到一半发现没数据库表，回头补。补完发现 API 接口签名又变了，测试也白写了。

问题有两个层面：

1. **任务之间有先后依赖，TodoWrite 没有表达依赖的能力**
2. **列表在内存里，会话结束就没了**

需要的不只是"待办清单"，而是一个**有依赖关系、持久化到磁盘、跨会话可恢复的任务系统**。

---

### TodoWrite vs Task System
|      | TodoWrite (s05) | Task System (s12)            |
| ---- | --------------- | ---------------------------- |
| 定位   | 当前任务的执行清单       | 可恢复的任务系统                     |
| 存储   | 进程内存            | `.tasks/{id}.json` 每个任务一个文件  |
| 依赖   | 无               | `blockedBy` / `blocks` 双向依赖图 |
| 生命周期 | 会话结束即消失         | 跨会话保留                        |
| 分工   | 没有"谁在做"的概念      | `owner` 认领机制                 |
| 粒度   | Agent 自己的步骤     | 可被认领、追踪、解锁的任务                |

两者在真实 Claude Code 中**同时存在**：交互式会话默认用 Task System，非交互式/SDK 默认用 TodoWrite，通过 feature flag 区分。

---

### Task 数据结构

每个任务是一个 JSON 文件，核心字段：


- **`id`**：唯一标识
- **`subject`**：简短标题
- **`status`**：`pending` → `in_progress` → `completed`
- **`blockedBy`**：此任务依赖的上游任务 ID 列表
- **`blocks`**：被此任务阻塞的下游任务 ID 列表
- **`owner`**：当前认领此任务的 Agent

真实 Claude Code 中还有 `activeForm`（进行时态，在 spinner 显示"正在做什么"）、`metadata`（扩展键值对），以及用 `.highwatermark` 文件防 ID 重用。

---

### 状态机：两个动作，三个状态

```
pending ──claim──→ in_progress ──complete──→ completed
```

`claim` 和 `complete` 是动作，`pending` / `in_progress` / `completed` 是状态。

**claim（认领）**：Agent 做一个任务前，先调用 claim。系统做两件事：检查所有 `blockedBy` 是否已完成（`can_start` 检查），检查是否已被别人认领（`owner` 冲突检测），都没问题就设置 `owner` 并切换到 `in_progress`。

**claim 的前置检查 `can_start`**：遍历 `blockedBy` 列表，任何一个依赖不是 `completed` 就返回 false。不存在的依赖也视为 blocked——防止引用错误 ID 时崩溃。如果 `blockedBy` 为空（无依赖），直接返回 true。

**complete（完成）**：任务做完后设为 `completed`，同时扫描所有其他 pending 任务，找出**刚刚被解锁**的下游任务——即那些 `blockedBy` 全部变成 `completed` 的任务，一并告知 Agent "这些现在可以做了"。

---

### 依赖图的威力

以"搭数据库、写 API、加测试"为例：

```
schema ──→ endpoints ──→ tests
   │
   └──→ docs
```

- `endpoints` 的 `blockedBy = [schema]`：schema 没做完前不能开始
- `tests` 的 `blockedBy = [endpoints]`：API 没写完不能写测试
- `docs` 的 `blockedBy = [schema]`：和 endpoints 可以并行，都只依赖 schema

当 schema 完成后，`endpoints` 和 `docs` 同时解锁。当 endpoints 完成后，`tests` 解锁。

这个依赖图之所以能跨会话，是因为每个任务的 JSON 文件都留在 `.tasks/` 目录下。即使会话结束，下次启动 Agent 读目录就能恢复完整状态：哪些做完了、哪些还在等、哪些被认领了。

### 并发认领的锁机制（real CC）

真实 Claude Code 中，多 Agent 同时认领任务时用**双重锁**防竞争：

1. **任务文件锁**：`proper-lockfile` 锁住 `{taskId}.json`，锁内重新读取（防 TOCTOU）、检查已认领/已完成/被阻塞，再设置 owner
2. **列表级锁**：`.lock` 文件，原子性扫描，检查同一个 Agent 是否已有其他进行中的任务
