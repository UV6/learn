长任务比如说 npm install，cc 会将其放到后台运行，等到下一轮以通知的形式注入。cc 怎么知道哪些模型要放到后台？两种判断形式：1. bash 里有一个参数来表示是否在后台运行，这样模型就可以自己判断。如果模型没有指定，就按照关键词 install deploy build test 等来判断

### 核心问题

用洗衣机的比喻：你不会站在洗衣机前干等 30 分钟。Agent 的 bash 工具也一样——`pip install torch` 要 10 分钟，`npm run build` 要 3 分钟。同步模式下 Agent 只能干等 bash 返回，而 LLM 按 token 计费，空转就是浪费。

不是所有命令都需要后台——读文件是毫秒级，`git status` 秒级返回。但 `npm install`？分钟级。

### 解决方案：慢操作丢后台，完成后注入通知

两种判断机制，显式请求优先：

- **模型显式请求**：bash 工具 schema 里增加了 `run_in_background: boolean` 参数，模型自己判断哪些命令需要后台跑
- **启发式兜底**：如果模型没指定，用关键词检测命令中是否包含 `install`、`build`、`test`、`deploy`、`compile` 等

### 同步 vs 后台

|          | 同步 (s12)  | 后台 (s13)                        |
| -------- | --------- | ------------------------------- |
| 慢操作处理    | Agent 干等  | daemon 线程执行                     |
| Agent 状态 | 阻塞        | 继续跑循环                           |
| 结果返回     | 命令结束时立即返回 | 下轮以通知形式注入                       |
| 判断依据     | —         | 模型 `run_in_background` 参数，启发式兜底 |

### 工作流程

**启动**：`start_background_task(block)` 把工具调用包装成 worker 函数，扔到 daemon 线程里。返回一个 `bg_id`，同时给 LLM 一个占位 tool_result：

```
> bash "npm install" (run_in_background=true)
→ start_background_task → bg_0001
→ tool_result: "[Background task bg_0001 started] Result will be available when complete."
```

LLM 看到这个占位就知道命令还在跑，可以先干别的。

**daemon=True** 确保 Agent 进程退出时后台线程也跟着退出。教学版用内存字典追踪状态，真实 CC 用文件重定向输出。

**收集**：`collect_background_results()` 每轮开始时扫描已完成的后台任务，格式化为 `<task_notification>` XML 通知：

```xml
<task_notification>
  <task_id>bg_0001</task_id>
  <status>completed</status>
  <command>npm install</command>
  <summary>added 245 packages in 45s</summary>
</task_notification>
```

通知和当前轮的工具结果**合并成同一条 user 消息**——LLM 在同一轮同时看到后台结果和前台的快速操作结果。

### 一个具体的执行序列

```
Turn 1:
  LLM → bash "npm install" (run_in_background=true)
  → start_background_task → bg_0001
  → tool_result: "[Background task bg_0001 started]..."
  → LLM: "OK, I'll install dependencies in background. Let me also read the config."

Turn 2:
  LLM → read_file "package.json" (fast, sync)
  → tool_result: file content
  → collect: bg_0001 done! inject <task_notification>
  → LLM 在同一条 user 消息里看到: package.json 内容 + npm install 完成通知
```

Agent 没干等——npm install 跑后台的时候，它去读了配置文件。

### 关键设计决策

**为什么通知不复用原始的 `tool_use_id`？** 因为原 tool call 已经用占位 `tool_result` 回复了。Messages API 要求一个 `tool_use` 只对应一个 `tool_result`。后台完成是独立事件，用独立的 `<task_notification>` 格式注入。

**为什么通知和 tool_result 合并在同一条 user 消息？** 这让 LLM 在一轮里同时看到新工具结果和后台完成通知，减少额外的轮次。

### real Claude Code 的复杂度

**没有真正的线程**：CC 运行在 Node.js/Bun 单线程事件循环中，"后台"只是"不 await"。Shell 进程的标准输出重定向到文件，进程独立运行。

**七种后台任务类型**：`local_bash`、`local_agent`、`remote_agent`、`in_process_teammate`、`local_workflow`、`monitor_mcp`、`dream`，每种有独立的注册和生命周期。

**停滞看门狗**：后台 bash 任务定期检查输出是否停滞——45 秒无增长后检测交互式提示（`(y/n)` 等），防止后台任务卡在无人响应的对话框。

**pendingToolUseSummary**：CC 在每批工具执行完后，启动一个 Haiku side-query（轻量模型）生成工具使用摘要，在主模型流式生成期间并行完成，给 SDK 做移动端进度展示。这也是"后台不阻塞主流程"思想的体现。
