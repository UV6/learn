### 核心问题

闹钟不需要你盯着它才会响。你设好 7:00，到点它自己响。s13 让 Agent 能后台执行慢操作，但所有操作仍然是你手动触发的——你说一句，Agent 动一下。"每天早上 9 点跑测试"、"每 30 分钟检查 CI 状态"，这些周期性任务不应该需要人每次来推。

### 解决方案：调度与执行解耦

Cron 调度分四层，各司其职：

```
Scheduler（调度线程）→ Queue（cron_queue）→ Queue Processor（交付线程）→ Consumer（agent_loop）
```

|层|职责|何时运行|
|---|---|---|
|**Scheduler**|daemon 线程，每秒轮询，判断时间到了没有|独立运行，不依赖 agent_loop|
|**Queue**|`cron_queue` 列表|调度线程写入已触发任务|
|**Queue Processor**|发现队列非空且 Agent 空闲，拉起 agent_loop|独立线程，200ms 轮询|
|**Consumer**|agent_loop 从队列消费，注入到 messages|被 queue processor 拉起，或被用户手动触发|

四层通过 `cron_queue`、`cron_lock`、`agent_lock` 解耦——生产者不关心谁是消费者，消费者不关心任务怎么产生的。

### 手动触发 vs 定时触发

|       | 手动触发 (s13) | 定时触发 (s14)      |
| ----- | ---------- | --------------- |
| 触发者   | 用户输入       | 调度线程            |
| 触发时机  | 随时         | cron 表达式指定      |
| 需要人参与 | 是          | 否，调度器自动入队       |
| 持久性   | —          | durable 任务跨重启保留 |

### CronJob 数据结构

每个定时任务包含五个字段：

- **`cron`**：标准五段式表达式（分钟 小时 日 月 星期）
- **`prompt`**：触发时注入给 Agent 的消息内容
- **`recurring`**：`true` 周期重复，`false` 一次性
- **`durable`**：`true` 写磁盘（`.scheduled_tasks.json`），跨会话保留；`false` 只在内存中
- **`id`**：唯一标识

Cron 表达式示例：

```
*/5 * * * *    每 5 分钟
0   9 * * *    每天早上 9:00
0   9 * * 1-5  工作日早上 9:00
30 14 28 2 *   2 月 28 日下午 2:30（一次性）
```

---

### 调度线程的关键设计

**独立于 agent_loop**：调度器跑在独立的 daemon 线程中，每秒轮询。即使 agent_loop 没在跑，调度器也在后台检查时间。进程关闭，调度也停（`daemon=True`）。

**date-aware minute_marker**：用 `"YYYY-MM-DD HH:MM"` 记录上次触发时间，而不是简单的 `"HH:MM"`。防止同一分钟重复触发，同时不会在第二天跳过本该触发的任务。

**单 job try/except**：每个 job 的匹配和触发放在独立的 try/except 里——一个坏 job 不会拖垮整个调度线程。

**一次性任务自动清理**：触发后从 `scheduled_jobs` 中删除，durable 的同时更新 `scheduled_tasks.json`。

### 执行流程：从时间到行动

```
1. 启动时：
   load_durable_jobs() → 从 .scheduled_tasks.json 恢复持久化任务
   Thread(cron_scheduler_loop) → 每秒轮询
   Thread(queue_processor_loop) → 200ms 检查队列

2. 用户注册：
   schedule_cron(cron="*/2 * * * *", prompt="run date", durable=True)
   → CronJob 写入内存 + .scheduled_tasks.json

3. 每 2 分钟：
   调度线程 → cron_matches → cron_queue.append(job)
   → queue processor 发现 Agent 空闲 → 获取 agent_lock
   → agent_loop → consume_cron_queue → 注入 "[Scheduled] run date"
   → LLM 收到消息，执行 date 命令

4. 关闭进程：
   调度线程跟着停（daemon=True）
   .scheduled_tasks.json 还在磁盘上
   下次启动 → load_durable_jobs → 任务恢复
```

---

### Durable vs Session-only

- **Durable**：任务定义持久化到 `.scheduled_tasks.json`。Agent **重启**后加载文件，恢复任务
- **Session-only**：只在当前进程内存中存活，Agent 关闭就没了

**重要前提**：cron 调度器必须在 Agent 进程内跑。进程关闭，调度也停。Durable 只意味着任务定义跨重启保留，下次 Agent 启动时调度器才会发现"该触发了"。如果需要"即使应用关闭也能定时跑"，应该用系统 crontab 或 systemd timer。

---

### 校验防线

`schedule_job` 在注册前校验 cron 表达式的合法性。从磁盘加载 durable job 时也会跳过非法表达式。这道防线确保：即使在磁盘文件被手动编辑损坏的情况下，调度器也不会被一个坏表达式拖垮。

---

### real Claude Code 的额外机制

**抖动防惊群**：重复性任务有基于任务 ID 确定性哈希的触发延迟（最多周期 10%，上限 15 分钟），防止大量任务在同一时刻精确触发。一次性任务落在 `:00` 或 `:30` 时最多提前 90 秒触发，分散负载。

**作业数上限**：`MAX_JOBS = 50`，超限返回错误，防止任务列表无限膨胀。

**自动过期**：重复性任务 7 天后自动过期（可配置，上限 30 天），过期前最后一次触发。

**文件锁**：`.scheduled_tasks.lock` 防止同一项目的多个 session 重复触发同一个 durable 任务。
