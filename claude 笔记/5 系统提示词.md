### 分段 + 按需拼接 + 缓存
把硬编码的一大段拆成独立的 **section**，每个 section 是一个主题。运行时根据真实状态决定加载哪些 section，拼接结果缓存起来避免重复计算。

四个 section，两种加载策略：

|Section|加载策略|判断依据|
|---|---|---|
|**identity**|始终加载|始终存在——你是谁、怎么做事|
|**tools**|始终加载|`enabled_tools`——当前可用工具有哪些|
|**workspace**|始终加载|始终存在——工作目录在哪|
|**memory**|按需加载|`.memory/MEMORY.md` 是否存在|

关键设计：section 是否加载取决于**真实状态**（工具是否存在、文件是否存在），不是在消息里搜关键词。

### 怎么做会话缓存的？
同一个 context 在同一轮对话中被多次问到时（比如多次 LLM 调用），重新拼接是浪费。用 JSON 序列化 context 生成稳定的 key，命中缓存直接返回。
但是这里的缓存只是防止重复拼接，和 cc 的 api prompt cache不是一回事。CC 的 API 层缓存通过 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 把 prompt 分为静态和动态两部分——静态部分跨请求复用 API 缓存，动态部分随 session 变化。教学版的缓存逻辑更简单，但方向一致：**把稳定和变化分开**。
