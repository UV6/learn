简单来说：通过把一个复杂任务拆成若干个子任务，然后维护一个带状态的列表：还没开始做，正在做，已做完。
### 核心问题

给 Agent 一个复杂任务："把所有 Python 文件改成 snake_case 命名，然后跑测试，修好失败。"

Agent 开始干活，改了 3 个文件，跑了测试，发现 2 个失败，开始修。修着修着，忘了最初的目标是"改成 snake_case"——测试失败把注意力全吸走了。

对话越长越严重：中间步骤的工具结果不断填满上下文，初始目标和剩余步骤被稀释。一个 10 步重构，做完 1-3 步就开始即兴发挥，因为 4-10 步已经被挤出注意力了。

### 解决方案：先列步骤再动手

`todo_write` 本身不做任何实际工作——不能读文件、不能跑命令、不能写代码。它只是一个**纯规划工具**，让 Agent 在动手之前先把思路写成清单。

### 工作原理

**`todo_write` 工具**接收一个带状态的列表：

- `pending`：还没开始
- `in_progress`：正在做（同时只能有一个）
- `completed`：已做完

数据保存在当前进程内存中，退出后清空。每次调用时，终端显示进度：

```
## Current Tasks
  [▸] 1. 扫描所有 Python 文件，找到非 snake_case 命名     ← in_progress
  [ ] 2. 逐个文件改名，确保引用同步更新                    ← pending
  [ ] 3. 运行完整测试套件                                  ← pending
  [ ] 4. 修复失败的测试（如果有）                          ← pending
  [ ] 5. 验证所有改动通过 lint                             ← pending
```

这就是 `todo_write` 三种状态构成的工作流程。Agent 做完一个，改状态为 `completed`，然后引动下一个 `pending`。

**Nag reminder**（教学版机制）：连续 3 轮没有调用 `todo_write` 时，循环自动注入一条提醒：

```
<reminder>Update your todos.</reminder>
```

这模拟了"Harness 主动督促"——如果 Agent 被工具结果吸引走了，系统会把它拉回来更新计划。

### 关键洞察

`todo_write` 不给 Agent 增加任何**执行能力**。它只增加了一种能力：**规划能力**。一个更复杂真实的 TodoWrite V2 版本叫做 Task System（后面 s12 会讲），增加了文件持久化、依赖图、并发锁，在这里 v1 版本先理解最简单的列表模型。

### 工具定义的精巧之处

工具定义中没有复杂的逻辑，只有一个简单的 schema 约束：

- `content`：步骤描述（字符串）
- `status`：三种状态，`enum` 限制

这个约束本身就是 agent 被引导遵守协议的关键——它强迫 Agent 用结构化的方式表达"我要做什么"，而不是"我想到了什么"。
