总结
 1. 先把大结果落盘：比如模型一次读取了好几个大文件，就将其落库，并留出一个索引让模型知道完整的内容在哪里
 2. 裁掉无关对话：只保留初始对话和最新的一些对话，用一个占位标记进行替代，超过阈值时进行裁剪
 3. 把工具的输出替换成占位符：1 是把读取的文件落库，这里是把工具的输出落库
 4. 前三个跑完之后，仍超过阈值，就把会话保存jsonl （json line）文件里，然后用一次 api 调用对会话进行压缩。然后用摘要替换原来的文件。

# 上下文管理：四层压缩管线

> 来源：Claude Code 源码 `compact.ts` / `autoCompact.ts` / `microCompact.ts` / `query.ts`，教学版 s08 做了简化。
>
> 核心原则：**便宜的先跑，贵的后跑。** 能改文本就不删整条，能删整条就不调 LLM。

---

## 背景

Agent 在运行过程中，每一轮对话的消息都会堆积在 `messages[]` 列表里。读过的文件内容、跑过的命令输出，全都在里面。上下文窗口是有限的，满了之后 API 直接拒绝请求。

压缩要做的事情就是：**在不丢失关键信息的前提下，持续给上下文腾地方。**

---

## 整体结构

压缩分成四层，按执行顺序排列。前三层是**纯文本操作**，不需要调用模型；第四层是 LLM 摘要，代价最高但在复杂场景下不可替代。

```
消息列表
    │
    ▼
┌─ L3: 大结果落盘  ───── 0 次 API ─┐
└─ L1: 裁旧对话    ───── 0 次 API ─┤  每轮自动执行
└─ L2: 旧结果占位  ───── 0 次 API ─┘
    │
    ├─ token 够了? ─── 是 ──→ 直接调 LLM
    │
    └─ 还不够? ─────── 是 ──→ L4: LLM 全量摘要（1 次 API）
                                 │
                                 ▼
                             调用 LLM
                                 │
                    API 仍然报错? ── 是 ──→ 应急裁剪（1 次 API）
```

三种触发模式，代价递增：

| 模式 | 触发条件 | 代价 |
|------|----------|------|
| ① 每轮自动 | 无条件，每次 LLM 调用前执行 | 0 API |
| ② 条件触发 | 前三层跑完 token 仍超阈值 | 1 API |
| ③ 异常触发 | API 仍然返回 `prompt_too_long` | 1 API + 更激进的裁剪 |

---

## L3 — 大结果落盘

### 位置

执行顺序的第一位。

### 做什么

统计最后一轮对话里所有工具输出（`tool_result`）的总大小。如果超过阈值，就按大小排序，**从最大的开始落盘**——把完整内容写入磁盘文件，上下文里只留下一个标记和一小段预览。

### 为什么

这是为了解决**单次工具调用的结果过大**的问题。比如模型一次性读了好几个大文件，即使只有这些结果本身，也可能直接打满上下文窗口。L3 在最早的时机把最大的内容搬出上下文，但保留一个索引标记让模型知道完整内容在哪，需要时可以重新读取。

### 关键点

- 只作用于**当前轮**的工具输出，不碰历史消息
- 计算结果总大小时按字节计，超过阈值才触发
- 按大小排序，优先落盘最大的，直到总量回到阈值以下
- 落盘的内容没有丢失，模型可以从磁盘重新读取
- **必须最先执行**：因为后续的 L2 会把旧结果替换成占位符，L3 必须在完整内容还在的时候完成落盘

---

## L1 — 裁掉无关的旧对话

### 位置

执行顺序的第二位。

### 做什么

当消息总数超过一定数量后，**保留开头的几条消息和末尾的大部分消息，中间的全部裁剪掉**，用一个占位标记代替。

### 为什么

Agent 跑了很久之后，最前面的对话（比如"帮我创建一个 hello.py"）和当前工作几乎无关，但全都占着位置。L1 做的是消息级别的粗粒度裁剪——把中间那段既不提供初始上下文、也不是当前工作的内容一刀切掉。

### 关键点

- 保留头部消息是为了保留**初始上下文**（用户最早的指令、约束等）
- 保留尾部是为了保留**当前工作上下文**
- 策略是"保两头、裁中间"
- 只在消息数超过阈值时触发

---

## L2 — 旧工具结果占位

### 位置

执行顺序的第三位。

### 做什么

遍历所有消息中的工具输出，**只保留最近几条的完整内容**，更旧的替换为一句话占位符。
L2 在消息内部，把旧的工具输出内容替换成占位符。旧的被压缩、新近的保留。

### 为什么

L1 裁掉了整条消息，但剩下的消息里，工具结果的内容仍然在累积。Agent 十分钟前读的文件内容还完整地躺在上下文里，早就不需要了，但每条占据着几百到几千字符。L2 做的是**工具结果级别的细粒度压缩**——不删消息本身，只把过时内容换成"如果还需要可以重新执行"的标记。

### 关键点

- 作用于所有 tool_result，只看"新不新"，按位置判断
- 保留最近 N 条完整，其余变占位符
- 真实 Claude Code 中通过 API 层的缓存编辑来实现，不会破坏 prompt cache 的前缀结构
- 只对部分工具生效（读文件、搜索、运行命令等产生大量输出的工具）

---

## L4 — LLM 全量摘要

### 位置

前三层跑完之后，如果估算 token 仍然超过阈值。

### 做什么

三步流程：

1. **保存 transcript**：把完整对话写入 `.transcripts/` 目录，JSONL 格式，保证信息不丢失
2. **调用 LLM 生成摘要**：把对话历史发给模型，要求保留关键信息
3. **替换消息列表**：所有旧消息被替换为一条摘要，模型从摘要继续工作

### 为什么

前三层都是纯文本或结构操作，不需要 API 调用，但代价是不能"理解"对话内容。它们可以删掉旧的、压缩大的，但无法判断"这段话重不重要"。在长时间、大范围的复杂工作中，各种任务信息和约束散布在整个对话历史里，前三层压缩无法完全解决问题时，就需要 LLM 来理解、筛选、重述整个对话。

### 关键点

- 这是唯一的 **1 次 API 调用**，代价最高，所以放在最后
- 摘要是对历史的浓缩，细节不再在上下文中
- 摘要需要保留：当前目标、重要发现、已改文件、剩余工作、用户约束
- **熔断器**：如果连续压缩失败 3 次，停止重试，避免白白消耗 API 调用

---

## 应急 — 被动压缩

### 位置

API 已经请求了，但返回了 `prompt_too_long` 错误。

### 做什么

比 L4 更激进——同样先保存 transcript 再生成摘要，但最终**只保留最后几条消息加上摘要**，用字节级精度裁剪到 API 可接受的大小。

### 为什么

有时候上下文增长的速度超过了压缩触发的速度，前三层预处理不够，L4 也没来得及或没拦住，API 直接报错了。这是最后的兜底手段。

### 关键点

- 只在 API 报错时触发，不走常规预处理流程
- 比 L4 更激进，裁剪比例更大
- 有重试上限，防止无限循环

---

## 核心设计思路

整个压缩管线围绕一个原则展开：**便宜的先跑，贵的后跑**。

前三层不花一分钱（0 API），先把能简单处理的空间释放出来。只有前三层实在不够时，才动用宝贵的 LLM 调用去做理解性的摘要。应急压缩则是最后的安全网，确保即使所有预防措施都失效，系统也不会崩溃。

这种层层递进的设计让 Agent 即使在超大项目中长时间工作，也不会因为上下文耗尽而中断。
