Lesson 0002: RAG 全流程

2026-06-28 · 预计时间 30 分钟 · 难度：中级

本课目标：理解 RAG（检索增强生成）的完整链路，用 LangChain 搭建文档加载 → 切片 → 向量化 → 检索 → 生成的端到端管道。

1. 为什么需要 RAG？

面试高频开场题：「你为什么要用 RAG？」

LLM 有两个硬伤：

  • 知识截止：训练数据停在某个时间点，不知道之后的事
  • 幻觉：遇到不知道的会编造答案，而不是说「我不知道」
  • 私有知识：不知道你公司的内部文档、产品手册、代码库

RAG 的解决思路：先检索相关上下文，再让 LLM 基于上下文回答。

面试标准答案：RAG = Retrieval（检索） + Augmented（增强） + Generation（生成）。核心思想是：把外部知识注入到 LLM 的 context window 中，让模型基于实际文档回答，而非依赖训练记忆。这既解决了知识截止问题，也显著降低了幻觉。

2. RAG 的五个环节

RAG 分为离线索引阶段（Indexing）和在线查询阶段（Query）：

┌───────────── Indexing (离线) ─────────────┐
│                                              │
│  文档 ──→ Load ──→ Split ──→ Embed ──→ Store │
│                                              │
└──────────────────────────────────────────────┘

┌─────── Query (在线) ───────┐
│                              │
│  问题 ──→ Embed ──→ Retrieve │
│                    ↓         │
│              Context + Prompt│
│                    ↓         │
│              LLM → Answer    │
│                              │
└──────────────────────────────┘

五个环节对应五个 LangChain 组件：

  1. Document Loader — 加载各种格式的文件（PDF、TXT、网页、数据库…）
  2. Text Splitter — 把长文档切成小块（chunk），控制大小和重叠
  3. Embedding Model — 把文本变成向量（数字），语义相似的文本向量也相近
  4. Vector Store — 存储和检索向量（ChromaDB、FAISS、Pinecone…）
  5. Retrieval Chain — 接收问题 → 检索相关文档 → 拼入 prompt → LLM 回答

3.3 完整代码

创建 rag.py：

import os
from dotenv import load_dotenv
from langchain.chat_models import init_chat_model
from langchain_community.document_loaders import TextLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_community.embeddings import HuggingFaceEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_core.prompts import ChatPromptTemplate
from langchain.chains.combine_documents import create_stuff_documents_chain
from langchain.chains import create_retrieval_chain

load_dotenv()

# ===== 离线阶段：构建索引 =====

# 1. Load — 加载文档
loader = TextLoader("knowledge.txt", encoding="utf-8")
docs = loader.load()
print(f"Loaded {len(docs)} document(s)")

# 2. Split — 切分成小块
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=200,       # 每块最大字符数
    chunk_overlap=40,     # 重叠字符数（避免截断上下文）
    separators=["\n\n", "\n", "。", ".", " "],  # 优先按段落切
)
chunks = text_splitter.split_documents(docs)
print(f"Split into {len(chunks)} chunks")

# 3. Embed + 4. Store — 向量化并存入向量库
embeddings = HuggingFaceEmbeddings(
    model_name="BAAI/bge-small-zh-v1.5",  # 中英文都支持的轻量模型
)
vectorstore = FAISS.from_documents(chunks, embeddings)
print("Vector store ready!")

# ===== 在线阶段：查询 =====

# 5. 设置 retriever
retriever = vectorstore.as_retriever(
    search_kwargs={"k": 3}  # 每次检索返回最相关的 3 个文档块
)

# 6. 初始化模型
model = init_chat_model(
    "openai:deepseek-chat",
    temperature=0,
    openai_api_key=os.environ["DEEPSEEK_API_KEY"],
    openai_api_base=os.environ["DEEPSEEK_BASE_URL"],
)

# 7. 构建 RAG prompt
system_prompt = """Use the following context to answer the question.
If you don't know the answer based on the context, just say you don't know.
Keep your answer concise.

Context:
{context}"""

prompt = ChatPromptTemplate.from_messages([
    ("system", system_prompt),
    ("human", "{input}"),
])

# 8. 创建 RAG chain
# step A: 把检索到的文档拼进 prompt
combine_docs_chain = create_stuff_documents_chain(model, prompt)
# step B: 先检索再拼接
rag_chain = create_retrieval_chain(retriever, combine_docs_chain)

# 9. 提问
questions = [
    "CloudSync Pro 个人版多少钱？",
    "DataGuard 支持什么安全特性？",
    "公司后端用什么技术栈？",
    "创始人是谁？",  # 知识库没有的信息
]

for q in questions:
    print(f"\n{'='*50}")
    print(f"Q: {q}")
    result = rag_chain.invoke({"input": q})
    print(f"A: {result['answer']}")
    # 可以看到检索了哪些文档块
    print(f"\n[Sources used: {len(result['context'])} chunks]")

4. 核心概念拆解

4.1 Text Splitter 的参数

参数含义面试考啥？
chunk_size 每块最大长度（字符数） 太小丢失上下文，太大检索不准。经验值 200-1000，调参实验。
chunk_overlap 相邻块的重叠长度 防止关键信息恰好被切断。通常设为 chunk_size 的 10-20%。
separators 切分优先级 RecursiveCharacterTextSplitter 按此顺序尝试切分，先段落，再句子，再空格。
实践建议：没有"完美"的 chunk 参数。不同文档类型需要不同策略。代码文档可能要 separators=["\n\n", "\n", ". "]，法律文档可能要更大的 chunk_size。面试时提到"需要根据文档类型和下游任务做实验调参"就能加分。

4.2 Embedding 的本质

Embedding 就是把一段文本映射为一个高维向量（比如 384 维的浮点数数组）。

关键性质：语义越相近的文本，向量之间的距离越近。

  • "今天天气真好" 和 "阳光明媚的一天" → 向量距离近 ✓
  • "今天天气真好" 和 "数据库查询语句" → 向量距离远 ✓

这就是为什么我们能用「北京的天气怎么样？」作为查询，检索出包含「晴朗，25°C」的文档块——虽然字面上没有重叠，但语义接近。

4.3 Vector Store 选型

场景推荐备注
本地开发/原型ChromaDB / FAISS零配置，内存/本地文件
中等规模生产Pinecone / Qdrant云托管，高性能
已有 PostgreSQLpgvector复用现有 DB，减少架构复杂度
企业级（百万+文档）Elasticsearch / Milvus分布式的，混合检索（向量+关键词）

5. 面试常见追问

问题你的回答要点
RAG 和 Fine-tuning 的区别？ RAG 是给模型外挂知识库，不改模型权重；Fine-tuning 是修改模型权重注入知识。RAG 知识可随时更新、可追溯来源；Fine-tuning 知识内化但无法追溯。实际项目中常常结合使用。
怎么评估 RAG 的效果？ 两个维度：(1) Retrieval 质量—命中率、MRR；(2) Generation 质量—基于上下文的忠实度（faithfulness）和答案相关性。常用 RAGAS 评估框架。
检索不准怎么办？ 按顺序排查：chunk 大小不合适 → 试试不同 embedding 模型 → 加 reranker 对检索结果精排 → 混合检索（BM25+向量）→ 调整 prompt 让 LLM 更严格地依赖上下文。
什么时候需要 Reranker？ 当 embedding 检索不够精确时（通常 k>20 时明显）。Reranker 用交叉编码器对候选文档精排，成本更高但更准。流程：embedding 粗筛 20 条 → reranker 留 5 条 → 给 LLM。
前端交互中 RAG 怎么处理？ 可以用 agent.stream() 流式输出，让用户看到逐字生成。同时在前端展示引用的来源文档块，增加可信度。LangChain 的 create_retrieval_chain 返回的 result["context"] 可以直接用来展示引用。

6. 动手练习

🛠 练习：扩展你的 RAG

  1. 把 knowledge.txt 换成你自己的真实文档（简历、笔记、代码注释都可以），看 RAG 能否回答关于它的内容
  2. 修改 chunk_size 参数（500, 1000），观察对检索结果的差异
  3. 把 RAG 包装成一个 Tool，集成到第一课的 Agent 中。这样 Agent 既可以直接回答（有知识），又可以调 RAG 工具（查私有文档）
  4. 尝试聊天历史：用 create_history_aware_retriever 让检索考虑对话上下文

7. 关键 API 速记

API用途
TextLoader(path)加载纯文本文件
RecursiveCharacterTextSplitter(chunk_size, chunk_overlap, separators)智能切分文档
HuggingFaceEmbeddings(model_name=...)本地 embedding，无需 API
FAISS.from_documents(chunks, embeddings)构建向量索引
vectorstore.as_retriever(k=N)从向量库创建检索器
create_retrieval_chain(retriever, combine_chain)组装完整 RAG 链路
