Lesson 0003: LangGraph 工作流

2026-06-28 · 预计时间 40 分钟 · 难度：中级偏高

本课目标：理解 LangGraph 的核心理念，用 StateGraph 构建一个有状态、多步骤、支持条件分支的 Agent 工作流。

1. 为什么需要 LangGraph？

第一课的 create_agent() 已经很方便了——为什么还要学 LangGraph？

一句话：create_agent() 适合"单轮工具调用"场景，LangGraph 适合"多步骤、有状态、有条件分支"的复杂 Agent。

具体对比：

场景create_agent()LangGraph
问天气 → 调工具 → 回答 ✅ 完美 过度设计
先搜索 → 如果不满意再搜 → 整合结果 ❌ 难以控制 ✅ 天然支持
A→B→判断→C或D→汇总 ❌ 不可控 ✅ 显式定义
需要暂停等待人工审批 ❌ 不支持 ✅ interrupt()
多 Agent 协作（并行+汇总） ❌ 不支持 ✅ Subgraph

面试时如果你只能说一句："LangGraph 把 Agent 逻辑建模为有向图——节点是操作，边是流转，状态在节点间传递。这比 create_agent 的黑盒循环更可控、可调试、可扩展。"

2. 核心概念：StateGraph

LangGraph 的三个基础元素：

┌────────────────────────────────────┐
│           StateGraph                │
│                                     │
│   State: 在所有节点间共享的数据      │
│   Node:  一个执行单元（函数）        │
│   Edge:  节点间的流转规则            │
│         · 普通边 (edge):  A → B     │
│         · 条件边 (conditional):      │
│           A → 判断 → B 或 C 或 D    │
│                                     │
└────────────────────────────────────┘

State 是核心——它承载了整个工作流中积累的信息：用户输入、中间结果、工具调用记录、最终答案。

面试要点：State 是 LangGraph 和普通函数调用的本质区别。每个节点只看到和修改 State，不需要知道前一个节点是谁。这实现了节点间的解耦和工作流的可组合性。

3. 动手：一个带自检的研究 Agent

场景：用户提问 → Agent 搜索 → 评估结果是否足够 → 不够就再搜 → 够了就生成答案。

创建 research_agent.py：

import os
from typing import TypedDict, Literal, Annotated
from dotenv import load_dotenv
from langchain.chat_models import init_chat_model
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langchain_core.messages import HumanMessage, AIMessage

load_dotenv()

# ============================================================
# Step 1: 定义 State — 工作流的"共享内存"
# ============================================================
class ResearchState(TypedDict):
    messages: Annotated[list, add_messages]  # 完整对话历史
    question: str                             # 原始问题
    search_count: int                         # 已搜索次数
    search_results: list[str]                 # 搜索结果列表
    final_answer: str                         # 最终答案
    next_action: str                          # 下一步：search | answer | complete

# ============================================================
# Step 2: 初始化模型
# ============================================================
model = init_chat_model(
    "openai:deepseek-chat",
    temperature=0,
    openai_api_key=os.environ["DEEPSEEK_API_KEY"],
    openai_api_base=os.environ["DEEPSEEK_BASE_URL"],
)

# ============================================================
# Step 3: 定义节点函数 — 每个函数是一个处理步骤
# ============================================================

def decide_node(state: ResearchState) -> dict:
    """决策节点：判断是否需要搜索"""
    question = state["question"]

    # 简单规则判断（实际项目中可用 LLM 判断）
    needs_search_keywords = ["什么", "怎么", "为什么", "多少", "谁", "哪里", "如何"]
    needs_search = any(kw in question for kw in needs_search_keywords)

    if needs_search and state["search_count"] == 0:
        return {"next_action": "search"}
    elif state["search_count"] >= 3:
        # 最多搜索 3 次，防止无限循环
        return {"next_action": "answer"}
    else:
        return {"next_action": "answer"}


def search_node(state: ResearchState) -> dict:
    """搜索节点：模拟一次搜索"""
    question = state["question"]
    count = state["search_count"] + 1

    # 模拟搜索结果（实际项目调搜索引擎 API）
    mock_knowledge = {
        "什么": "LangChain 是一个用于构建 LLM 应用的框架，"
                "核心抽象包括 Chain、Agent、Tool。"
                "它支持 Python 和 JavaScript。",
        "怎么": "使用 LangGraph 构建 Agent 的步骤："
                "1) 定义 State 2) 定义 Node 函数 "
                "3) 构建 Graph 4) 编译运行。"
                "关键是用条件边控制流转。",
        "为什么": "LangGraph 相比 create_agent() 的优势："
                "显式状态管理、条件分支、Human-in-the-Loop、"
                "子图（多Agent协作）。它把 Agent 循环从黑盒变为白盒。",
    }

    # 匹配关键词获取结果
    result = ""
    for key, value in mock_knowledge.items():
        if key in question:
            result = value
            break
    if not result:
        result = f"(第{count}次搜索) 未找到相关信息。"

    return {
        "search_count": count,
        "search_results": state["search_results"] + [result],
    }


def answer_node(state: ResearchState) -> dict:
    """回答节点：基于搜索结果生成最终答案"""
    results = "\n".join(
        f"- {r}" for r in state["search_results"]
    ) if state["search_results"] else "（无需搜索，直接回答）"

    prompt = f"""基于以下信息回答用户问题。

用户问题：{state["question"]}

搜索到的信息：
{results}

请用中文给出准确、简洁的回答。如果信息不足，请诚实说明。
"""

    response = model.invoke([HumanMessage(content=prompt)])
    return {"final_answer": response.content}


# ============================================================
# Step 4: 定义路由函数 — 条件边的"判断逻辑"
# ============================================================
def route_after_decide(state: ResearchState) -> Literal["search", "answer"]:
    """决定从 decide 节点去 search 还是 answer"""
    return state["next_action"]


def route_after_evaluate(state: ResearchState) -> Literal["search", "answer"]:
    """决定从 evaluate 节点去 search（信息不够）还是 answer（够了）"""
    if state["search_count"] < 2 and state["search_count"] > 0:
        # 只搜了一次，可能不够，再搜一次
        return "search"
    return "answer"


# ============================================================
# Step 5: 构建 Graph
# ============================================================
builder = StateGraph(ResearchState)

# 添加节点
builder.add_node("decide", decide_node)
builder.add_node("search", search_node)
builder.add_node("answer", answer_node)

# 添加边
builder.add_edge(START, "decide")

# 条件边：decide 可能去 search 或 answer
builder.add_conditional_edges(
    "decide",
    route_after_decide,
    {
        "search": "search",
        "answer": "answer",
    }
)

# 条件边：search 后评估是否还要继续搜
builder.add_conditional_edges(
    "search",
    route_after_evaluate,
    {
        "search": "search",   # 循环回去再搜
        "answer": "answer",
    }
)

builder.add_edge("answer", END)

# 编译
graph = builder.compile()

# ============================================================
# Step 6: 运行测试
# ============================================================
test_questions = [
    "什么是 LangChain？",       # 需要搜索
    "今天天气真好",             # 不需要搜索
    "为什么用 LangGraph？",     # 需要搜索
]

for q in test_questions:
    print(f"\n{'='*60}")
    print(f"❓ {q}")

    result = graph.invoke({
        "messages": [],
        "question": q,
        "search_count": 0,
        "search_results": [],
        "final_answer": "",
        "next_action": "",
    })

    print(f"🔍 搜索了 {result['search_count']} 次")
    print(f"📝 答案：{result['final_answer'][:200]}...")

4. 代码走读

4.1 State 定义

State 用 TypedDict 定义，Annotated[list, add_messages] 是 LangGraph 的特殊语法——表示 messages 字段不是"覆盖"而是"追加"：

class ResearchState(TypedDict):
    messages: Annotated[list, add_messages]  # 追加模式
    question: str                             # 覆盖模式（默认）

面试考点："你怎么管理 State 的字段更新策略？" 答：普通字段默认覆盖，对话消息用 add_messages reducer 做追加。

4.2 条件边 vs 普通边

类型API何时用
普通边 builder.add_edge(A, B) A 完成后一定去 B
条件边 builder.add_conditional_edges(A, router, {"x": B, "y": C}) A 完成后根据 State 决定去哪

router 函数接收 State，返回一个字符串——这个字符串必须匹配 add_conditional_edges 第三个参数中的某个 key。

4.3 循环控制

本课代码中 search → evaluate → search → evaluate → ... 形成了循环。LangGraph 不会让这个循环无限运行——它在每次迭代后做判断。你必须自己加终止条件（如 search_count >= 3），否则 LangGraph 有默认的递归上限。

注意：LangGraph 的循环是 同步执行 的——每次循环都是一个新的 LLM 调用。如果循环次数太多，延迟会线性增长。面试时提到"需要设置合理的递归上限和 token 预算"能加分。

5. 面试高频追问

问题你的回答要点
LangGraph 和 create_agent 什么关系？ create_agent 底层就是用 LangGraph 实现的，返回的是一个预配置好的 StateGraph。LangGraph 是引擎，create_agent 是快捷方式。
State 可以用 Pydantic 吗？ 可以，推荐用 Pydantic BaseModel 替代 TypedDict（有验证、默认值、序列化支持）。LangGraph 同时支持两种。
怎么处理并行执行？ 用 builder.add_node("parallel", [node_a, node_b]) 给一个节点传函数列表，LangGraph 并行执行它们，结果合并到 State。
怎么暂停等待人工？ 用 interrupt() 函数暂停图执行，等人工确认后用 Command(resume=...) 恢复。这是 LangGraph 的 Human-in-the-Loop 机制。
Checkpointing 是什么？ LangGraph 的"快照机制"。每次状态变更自动保存，支持时间旅行（回到任意历史状态）、错误恢复、人工审批断点续传。用 MemorySaver 或 SqliteSaver 实现。

6. 动手练习

🛠 练习：扩展你的 Graph

  1. 加一个 evaluate_node：在 search 和 answer 之间加一个 explicit evaluate 节点，用 LLM 判断搜索结果是否足以回答问题，不够就触发新一轮 search
  2. 集成第一课的 Tool：把 weather agent 的 get_weather 工具作为 LangGraph 的一个 node
  3. 加 Checkpointing：用 MemorySaver 保存图执行历史，体验 graph.get_state() 查看中间状态
  4. 并行搜索：把 search_node 改为并行执行两个不同的搜索（如 Wikipedia + 本地知识库），汇总结果

7. 关键 API 速记

API用途
StateGraph(StateType)创建状态图，泛型参数为 State 类型
builder.add_node(name, func)添加节点（可传单个函数或函数列表做并行）
builder.add_edge(from, to)添加普通边（始终执行的流转）
builder.add_conditional_edges(from, router, mapping)添加条件边（根据 State 决定路径）
builder.compile(checkpointer=...)编译图，可选 checkpointer 实现状态持久化
graph.invoke(state_dict)执行图，返回最终 State
graph.stream(state_dict)流式执行，逐步返回每个节点的输出
Annotated[type, reducer]定义带 reducer 的 State 字段（如 add_messages）

8. LangGraph vs 其他编排框架

框架特点适合
LangGraph状态图模型、Checkpointing、Human-in-the-Loop复杂 Agent 工作流
LangChain create_agent封装好的 Agent 循环简单工具调用场景
CrewAI角色扮演多 Agent、高层抽象多角色协作但流程简单
AutoGen对话驱动多 Agent、微软出品多 Agent 对话协作
OpenAI Swarm轻量多 Agent 实验框架快速原型和实验
面试加分：提到 LangGraph 是"底层引擎"的概念——create_agent、CrewAI 都可以基于 LangGraph 实现。掌握了 LangGraph 就掌握了 Agent 编排的底层能力。
