Lesson 0004: Tools + Memory

2026-06-28 · 预计时间 40 分钟 · 难度：中级偏高

本课目标：深入 Agent 的两大核心——Tools（不只是函数声明，还包括错误处理、结构化输出、异步调用）和 Memory（短期记忆、长期记忆、LangGraph checkpoint 驱动的持久化记忆）。

Part A: Tools 深入

1. Tool 不只是函数

第一课我们定义 tool 只需要 type hints + docstring。面试官追问："这样够用吗？生产环境呢？"

LangChain 提供了三层 tool 定义方式：

# Level 1: 最简单 — 函数即 Tool
# LangChain 自动读 type hints + docstring 生成 schema
def get_weather(city: str) -> str:
    """Get current weather for a city.

    Args:
        city: City name
    """
    return f"Sunny, 25°C in {city}"

# 直接传给 create_agent
agent = create_agent(model, tools=[get_weather])
# Level 2: @tool 装饰器 — 精确控制 schema
from langchain_core.tools import tool

@tool
def search_database(
    query: str,
    limit: int = 5,
    include_archived: bool = False,
) -> list[dict]:
    """Search the internal product database.

    Args:
        query: SQL-like search query
        limit: Max results to return (1-20)
        include_archived: Whether to include archived products
    """
    # ... 实际数据库查询 ...
    return [{"id": 1, "name": "CloudSync Pro"}]

# @tool 允许设 name, description, args_schema
# 适合有复杂参数、默认值、可选参数的工具
# Level 3: BaseTool 子类 — 完整控制
from langchain_core.tools import BaseTool
from pydantic import BaseModel, Field

class RefundInput(BaseModel):
    order_id: str = Field(description="Order ID to refund")
    amount: float = Field(ge=0, le=99999, description="Amount in USD")
    reason: str = Field(description="Refund reason")

class RefundTool(BaseTool):
    name: str = "process_refund"
    description: str = "Process a refund for an order. Requires manager approval."
    args_schema: type[BaseModel] = RefundInput

    # 可注入依赖
    db_session: Any = None

    def _run(self, order_id: str, amount: float, reason: str) -> str:
        # 真实的退款逻辑
        return f"Refund {amount} for order {order_id}: approved"

# BaseTool 优势：
# 1. Pydantic 参数验证（Field 约束）
# 2. 依赖注入（db_session 等）
# 3. 可重写的错误处理钩子
# 4. 异步 _arun 支持
面试要点：三种方式的选择——原型用函数，复杂参数用 @tool，生产环境（需要验证、依赖注入、错误处理）用 BaseTool。

2. Tool 的错误处理

面试常见场景题："Tool 调用失败了怎么办？Agent 会崩溃吗？"

from langchain_core.tools import tool

@tool
def fragile_api(query: str) -> str:
    """Call external API. May fail."""
    try:
        result = call_external_service(query)
        return result
    except TimeoutError:
        return "API 超时，请稍后重试"
    except Exception as e:
        # 返回给 LLM 可理解的错误信息，而不是抛异常
        return f"工具调用失败：{str(e)}。请尝试其他方法。"

关键原则：Tool 永远不抛异常给用户。返回一个描述性错误字符串，让 LLM 决定怎么处理（重试、换工具、承认失败）。

常见错误：新手会在 tool 里面 raise Exception。Agent 拿到异常后行为不可预期。正确做法是 catch 所有异常，返回人类可读的错误信息。

@tool
async def slow_search_async(query: str) -> str:

Part B: Memory 深入

4. Memory 的三种形态

面试必问："Agent 怎么记住上下文？"

类型机制场景LangChain 实现
短期记忆 把历史对话放进 context window 多轮对话中记住前面的问题 messages 列表 + add_messages reducer
摘要记忆 LLM 生成对话摘要，丢弃原始历史 长对话节省 token SummarizationNode / LLM 调用
持久化记忆 存到外部存储，按需检索 跨 session 记住用户偏好 LangGraph Checkpointer + Vector Store
面试回答框架：说完三种记忆后加一句——"实际项目中最常见的组合是：会话语义上下文用 messages（短期）+ 跨 session 的用户偏好用向量库（长期）+ LangGraph MemorySaver 做 checkpoint（恢复中断）。"

5. 短期记忆：messages 列表

这是你在前几课不知不觉已经在用的：

# 多轮对话 — 把整个 messages 列表传入
messages = [
    {"role": "user", "content": "北京天气？"},
    {"role": "assistant", "content": "今天北京晴朗，25°C。"},
    {"role": "user", "content": "那昨天呢？"},  # LLM 看到上文，知道"昨天"指北京
]

result = agent.invoke({"messages": messages})

问题：context window 有限（DeepSeek 64K，GPT-4 128K），历史消息会填满。

解法：有两种常用策略——

策略 A: 滑动窗口

# 只保留最近 N 条消息
MAX_HISTORY = 20
recent_messages = messages[-MAX_HISTORY:]

策略 B: 摘要压缩

# 用 LLM 把旧消息压缩成摘要
def summarize_history(messages: list, model) -> str:
    prompt = f"用中文概括以下对话的关键信息：\n{messages}"
    return model.invoke(prompt).content

old_summary = summarize_history(messages[:-10], model)
# 新 prompt = 摘要 + 最近 10 条原始消息
context = f"对话历史摘要：{old_summary}\n\n最近对话：{messages[-10:]}"
面试加分：提到"摘要 + 滑动窗口的混合策略"——用 LLM 摘要压缩旧对话，同时保留最近 N 条原始消息。这是 DeepMind 论文中验证的最优方案，比单纯截断或纯摘要效果都好。

6. LangGraph Checkpoint 持久化记忆

本课最重要的概念：LangGraph 的 Checkpointer 是它区别于 create_agent() 的核心能力。

from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import StateGraph

# 编译时注入 checkpointer
checkpointer = MemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# 每次调用带 thread_id — 作为"会话标识"
config = {"configurable": {"thread_id": "user-123"}}

# 第一次对话
result1 = graph.invoke(state, config)
# 第二次对话（同 thread_id）— LangGraph 自动加载上次状态！
result2 = graph.invoke(state, config)
# result2 的 messages 包含 result1 的所有历史

Checkpointer 做了什么：

  1. 每次 invoke() 后自动保存完整的 State（messages、search_results 等）
  2. 下次同 thread_id 的调用自动恢复 State
  3. 支持 graph.get_state(config) 查看任意时间点的状态
  4. 支持 graph.get_state_history(config) 查看状态变更历史（时间旅行）

Checkpointer 选型

Checkpointer存储场景
MemorySaver内存 dict开发/测试/短 session
SqliteSaver本地 SQLite 文件单机持久化，重启不丢
PostgresSaverPostgreSQL生产环境，多实例共享
RedisSaverRedis高并发低延迟
面试问题："多轮对话怎么实现？" 标准答案分三层——1) 短期：messages 列表传入 context window 2) 长期：LangGraph Checkpointer 持久化 State，按 thread_id 恢复 3) 跨 session：用 Vector Store 存用户偏好，检索注入 context。

7. Human-in-the-Loop

Memory 的另一面：有些操作不能让 Agent 自动执行。

from langgraph.types import interrupt

def refund_node(state: AgentState) -> dict:
    """处理退款 — 需要人工审批"""
    order_id = state["pending_refund"]["order_id"]
    amount = state["pending_refund"]["amount"]

    # interrupt() 暂停执行，等待人工介入
    approval = interrupt(f"确认退款 ${amount}？订单 {order_id}")

    if approval == "approved":
        # 执行退款
        return {"refund_status": "done"}
    else:
        return {"refund_status": "cancelled"}

# 运行时：
# 1. graph.invoke(state) → 停在 refund_node，抛出 GraphInterrupt
# 2. 人工审查后调用：
#    graph.invoke(Command(resume="approved"), config)
# 3. 从断点继续执行
面试场景："Agent 要发邮件/退款/删数据，怎么加安全控制？" → 回答用 interrupt() + Command(resume=...) 的 Human-in-the-Loop 模式。这是 LangGraph 的核心卖点。

8. 全技能融合：一个完整的记忆增强 Agent

把前三课的所有概念组合在一起：

from typing import TypedDict, Annotated, Literal
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import interrupt
from langchain_core.tools import tool
from langchain.chat_models import init_chat_model

# ----- State -----
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]
    user_profile: dict       # 长期记忆：用户偏好（从向量库查的）
    pending_action: dict      # 需要人工审批的操作

# ----- Tools -----
@tool
def search_kb(query: str) -> str:
    """Search internal knowledge base."""
    return "Found: ..."

@tool
def send_email(to: str, body: str) -> str:
    """Send email. Requires manager approval."""
    return "Email queued"

# ----- Nodes -----
def agent_node(state: AgentState) -> dict:
    """LLM 推理节点 — 决定调用哪个 tool"""
    model_with_tools = model.bind_tools([search_kb, send_email])
    profile = state.get("user_profile", {})
    system = f"你是助手。用户偏好：{profile}"
    response = model_with_tools.invoke(
        [{"role": "system", "content": system}] + state["messages"]
    )
    return {"messages": [response]}

def tool_node(state: AgentState) -> dict:
    """执行 tool 调用"""
    last_msg = state["messages"][-1]
    # langgraph 内置的 ToolNode 比手动写更优雅
    # 这里展示手动逻辑以理解原理
    results = []
    for tc in last_msg.tool_calls:
        if tc["name"] == "send_email":
            # 高风险操作 → 走审批
            return {"pending_action": tc}
        else:
            results.append(execute_tool(tc))
    return {"messages": results}

def approve_node(state: AgentState) -> dict:
    """审批节点（Human-in-the-Loop）"""
    action = state["pending_action"]
    interrupt(f"Approve sending email to {action['to']}?")
    # ... 执行操作 ...
    return {"pending_action": None}

# ----- Graph -----
def should_continue(state: AgentState) -> Literal["tools", "approve", "end"]:
    last = state["messages"][-1]
    if state.get("pending_action"):
        return "approve"
    if hasattr(last, "tool_calls") and last.tool_calls:
        if last.tool_calls[0]["name"] == "send_email":
            return "approve"
        return "tools"
    return "end"

builder = StateGraph(AgentState)
builder.add_node("agent", agent_node)
builder.add_node("tools", tool_node)
builder.add_node("approve", approve_node)
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", should_continue)
builder.add_edge("tools", "agent")     # 拿到 tool 结果后回去推理
builder.add_edge("approve", "agent")   # 审批后回去推理

# 编译 — 带 Checkpointer 实现持久化记忆
graph = builder.compile(checkpointer=MemorySaver())

# 运行
config = {"configurable": {"thread_id": "user-42"}}
graph.invoke({"messages": [{"role": "user", "content": "给 ceo@corp.com 发邮件"}]}, config)

这个例子展示了：Tool 调用 → 条件分支 → 人工审批 → Checkpointer 记忆 → 长期状态管理。这是面试中能展示理解深度的完整案例。

9. 面试高频追问

问题你的回答要点
Tool 太多（50+）怎么选？ 1) 用 tool description 写好说明让 LLM 自选 2) 按领域分组，先路由到子领域再选 Tool 3) 用 Semantic Router 或 embedding 做预筛选
Context Window 满了怎么办？ 分层策略：摘要压缩旧历史 + 滑动窗口保留近期 + 关键信息存外部存储按需检索（如 faiss）+ 用 context window 管理工具（如 trim_messages）
怎么实现"记住用户偏好"？ user_profile 存入可持久化存储（DB/Redis），每次 session 启动时加载。关键信息（语言偏好、权限级别）注入到 system prompt。
多个用户并发怎么隔离记忆？ 用 thread_id 隔离。每个用户/会话一个 thread_id，Checkpointer 自动按 thread_id 存取状态。数据库隔离也由 thread_id 保证。
失败重试怎么设计？ Tool 层返回错误描述（不抛异常），LLM 决定重试策略。Graph 层可加 retry node 和计数 limit。LangGraph 的 Command(goto="retry") 可实现动态跳转。
