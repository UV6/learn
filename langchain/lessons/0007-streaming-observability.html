Lesson 0007: 流式输出与可观测性

2026-06-30 · 预计时间 35 分钟 · 难度：中级

本课目标：掌握 Agent 的 streaming 输出（从 agent.stream() 到自定义 streaming events），理解生产环境的可观测性体系（LangSmith tracing、token tracking、错误监控）。

1. 为什么需要 Streaming？

面试场景题："用户反馈 Agent 响应太慢（10 秒才出第一个字），怎么优化？"

答案：用 streaming 让用户看到逐字输出——感知延迟从 10 秒降到 0.5 秒。

但实际上 streaming 不止改善体验：

  • 中断控制：用户可以中途停止生成（看到方向不对就打断）
  • 节省成本：用户提前中断 → 不再生成后续 token → 省 token
  • 可观测性：流式事件暴露了内部状态——tool 调用时机、推理步骤

2. LangChain 的 Streaming 层次

层级API返回什么场景
Token 级 model.stream() 逐 token 的 AIMessageChunk 聊天 UI 打字机效果
Agent 级 agent.stream() 每个步骤的事件流 Agent 执行过程可视化
Graph 级 graph.stream() 每个节点的状态变更 LangGraph 工作流监控
事件级 graph.astream_events() 所有内部事件 自定义 UI、调试工具

3. 三种 Streaming 的实际代码

3.1 Token 级 Streaming

# 逐 token 输出
for chunk in model.stream("写一首关于AI的诗"):
    print(chunk.content, end="", flush=True)
# 输出：人 工 智 能 如 朝 阳 ， ...

# 在 Agent 中使用（需要先 stream_mode="messages"）
for chunk in agent.stream(
    {"messages": [{"role": "user", "content": "北京天气？"}]},
    stream_mode="messages"
):
    if isinstance(chunk, AIMessageChunk):
        print(chunk.content, end="", flush=True)

3.2 Agent 级 Streaming (stream_mode)

# stream_mode 控制返回粒度
# "values" — 每步的完整 State（默认）
# "updates" — 每步 State 的变更
# "messages" — 直接拿到 LLM token
# "custom" — 自定义事件

for chunk in agent.stream(input, stream_mode="updates"):
    # chunk 是 {node_name: node_output}
    for node_name, output in chunk.items():
        print(f"[{node_name}] {output}")
# 输出：
# [agent] {'messages': [AIMessage(tool_calls=[...])]}
# [tools]  {'messages': [ToolMessage("晴朗，25°C")]}
# [agent] {'messages': [AIMessage("北京今天天气晴朗...")]}

3.3 Graph 级 Streaming + 自定义事件

# LangGraph 的 stream 展示每个 node 的执行
for event in graph.stream(initial_state, config):
    # event = {node_name: state_update}
    node_name = list(event.keys())[0]
    print(f"→ 执行节点: {node_name}")
# 输出：
# → 执行节点: decide
# → 执行节点: search
# → 执行节点: answer
面试加分："除了改善用户体验，streaming 的另一大价值是让 Agent 的内部决策过程对用户透明。用户可以看到 Agent 在调用哪个 tool、思考了什么——这对建立信任至关重要。金融、医疗场景中，这甚至是合规要求。"

4. 可观测性体系

生产环境中，"Agent 为什么给出了这个答案？"——这个问题必须可回答。

维度工具回答什么问题
Tracing LangSmith, OpenTelemetry 这次调用的完整链路是什么？
Token Tracking LangSmith, callback handler 花了多少 token？哪个环节最费？
Error Tracking Sentry, LangSmith runs Tool 调用失败了多少次？为什么？
Cost Tracking LangSmith, custom callback 这次对话花了多少钱？
Quality Evaluation LangSmith eval, RAGAS 回答质量如何？有没有幻觉？
Latency Profiling LangSmith, custom timing 哪个环节最慢？LLM 推理？Tool 调用？

6. 自建监控：Callback Handler

如果不用 LangSmith，可以用 LangChain 的 callback 自己收集：

from langchain.callbacks import BaseCallbackHandler

class MetricsHandler(BaseCallbackHandler):
    def __init__(self):
        self.total_tokens = 0
        self.total_cost = 0
        self.tool_calls = 0
        self.llm_calls = 0

    def on_llm_end(self, response, **kwargs):
        self.llm_calls += 1
        tokens = response.llm_output["token_usage"]["total_tokens"]
        self.total_tokens += tokens

    def on_tool_end(self, output, **kwargs):
        self.tool_calls += 1

# 使用
handler = MetricsHandler()
agent.invoke(input, config={"callbacks": [handler]})
print(f"LLM calls: {handler.llm_calls}, Toolkit calls: {handler.tool_calls}")
print(f"Total tokens: {handler.total_tokens}")

7. 面试高频追问

问题你的回答要点
"怎么监控 Agent 的质量？" 分两层：(1) 技术指标——延迟 P50/P99、错误率、token 消耗、tool 调用成功率 (2) 质量指标——用户满意度（👍👎）、答案准确性（人工抽检或 LLM-as-judge）、幻觉率。用 LangSmith 的 annotation queue 做持续评估。
"Token 成本怎么优化？" 五招：1) 缩短 system prompt 2) 多 Agent 分层用不同模型（便宜的做路由）3) context window 管理（摘要压缩）4) 缓存常见问题的答案 5) streaming 让用户提前中断。
"LangSmith vs 自建？" LangSmith 是 LangChain 生态的一部分，集成成本为零。适合快速原型和中型项目。大规模部署时可能需要自建（成本、数据隐私），但架构参考 LangSmith 的设计——trace 树 + run metadata。
"怎么 debug Agent 的决策？" 三件套：(1) LangSmith trace 看完整调用链 (2) stream_mode="updates" 看每步的 State 变更 (3) 在关键 node 中打印 state 快照。面试时强调"可调试性比美观更重要"能加分。
