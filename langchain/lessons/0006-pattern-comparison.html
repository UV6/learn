Lesson 0006: 架构模式对比与选型

2026-06-30 · 预计时间 35 分钟 · 难度：中高级

本课目标：系统对比三种 Agent 架构模式，建立选型决策框架，能在面试中清晰回答"你选哪个方案，为什么？"

1. 三种模式的本质差异

2. 选型决策树

问题 1: 任务类型是否可预定义？
  ├── 是 (如：客服、技术支持、销售)
  │     └── 问题 2: 各任务间有严格的执行顺序？
  │           ├── 是 → Supervisor + 条件边序列
  │           └── 否 → Supervisor + 自由路由
  │
  └── 否 (如：用户随便问，Agent 自己判断)
        └── 问题 3: 单个 Agent 的 tool 数量是否 > 10？
              ├── 是 → Subagent 委托（避免 context 过载）
              └── 否 → 单 Agent (create_agent)

3. 各模式的代码复杂度对比

面试官可能让你现场设计："给你一个场景，怎么写代码？"——这是对你工程判断力的考查。

模式最少代码行数核心复杂度
单 Agent~20 行只需定义 tool + create_agent
Supervisor~80 行多 Agent node + router function + 循环控制
Subagent~50 行子 Agent 包装为 @tool + system prompt 设计
工程判断力：面试时先问自己——"这个场景用 create_agent 能不能搞定？"能就不要引入多 Agent。每次多一个 Agent，就多一套 system prompt、多一个延迟源、多一层调试复杂度。最简单的方案就是最好的方案，直到被证伪。

4. 混合架构：生产环境的真实形态

生产环境很少是"纯"某一种模式。最常见的是 Supervisor + Subagent 混合：

                    ┌──────────────┐
                    │  Supervisor  │
                    └──┬───┬───┬──┘
                       │   │   │
           ┌───────────┘   │   └───────────┐
           ↓               ↓               ↓
    ┌────────────┐  ┌────────────┐  ┌────────────┐
    │  客服 Agent │  │  订单 Agent │  │  技术 Agent │
    │  (单Agent)  │  │ (Subagent) │  │ (Supervisor │
    │             │  │            │  │   嵌套)     │
    │ tools:      │  │ tools:     │  │            │
    │ - search_kb │  │ - refund   │  │  ┌─Code    │
    │ - escalate  │  │ - lookup   │  │  ├─Search  │
    └────────────┘  │ - code_    │  │  └─Review  │
                    │   reviewer │  └────────────┘
                    └────────────┘

每层可以用不同的模型（成本优化）：

  • Supervisor — 便宜模型（GPT-4o-mini / haiku）只做路由
  • 客服 Agent — 便宜模型，响应快
  • 技术 Agent — 强模型（sonnet / GPT-4o），推理重
  • 子 Agent (code reviewer) — 强模型，安全要求高

5. 面试高频追问

问题你的回答要点
"你之前项目为什么选 XX 模式？" 用决策树回答——先描述场景特征（任务是否可预定义、tool 数量、安全要求），再映射到模式选择。展示你是有决策框架的，不是拍脑袋。
"如果 Supervsior 路由错了怎么办？" 三层防御：(1) 子 Agent 的 system prompt 中有边界检查——不属于自己领域的拒绝处理，返回给 Supervisor (2) Supervisor 设重试上限 (3) 加 fallback node——路由失败时直接给人或给通用 Agent。
"多 Agent 的延迟怎么优化？" (1) 并行 node（LangGraph 原生支持）(2) 子 Agent 用小模型 (3) Supervisor 缓存——相同类型的任务不用每次都决策 (4) 减少不必要的 Agent 间通信——子 Agent 直接输出结果，不需要层层汇报。
"Subagent 嵌套太深怎么办？" 设置最大深度限制。超过 2 层嵌套通常是设计问题——检查是否应该合并某些 Agent 或用 Supervisor 模式替代。生产环境一般不超过 2 层。
