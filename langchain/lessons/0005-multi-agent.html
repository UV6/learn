Lesson 0005: 多 Agent 协作

2026-06-28 · 预计时间 40 分钟 · 难度：中高级

本课目标：掌握多 Agent 协作的三种核心模式——Supervisor 路由、Subagent 委托、Swarm 协商——并理解每种模式的面试考察点和 LangGraph 实现方式。

1. 为什么需要多 Agent？

单 Agent 的瓶颈：

  • 上下文过载：一个 prompt 里塞 50 个 tool，LLM 的选择准确率急剧下降
  • 责任模糊：一个 Agent 既要查数据库、又要写代码、又要做客服——prompt 太长，容易串
  • 无法并行：单 Agent 串行调用 tool，复杂任务延迟高
  • 安全边界模糊：读写操作混在同一个 Agent，风险大
面试一句话："多 Agent 的核心思想是分治——把复杂任务拆成多个专业化子 Agent，每个只关注自己的领域。通过编排层协调它们的分工和通信。"

2. 三种多 Agent 模式

模式结构适用场景面试频率
Supervisor 路由 一个监督者 → 多个专业 Agent → 监督者汇总 客服分流、任务分类 ⭐⭐⭐⭐⭐
Subagent 委托 主 Agent 把子任务委托给子 Agent（子 Agent 作为 Tool） 复杂研究、代码生成+审查 ⭐⭐⭐⭐
Swarm/Peer 协商 多个平等 Agent 互相传递控制权 开放讨论、创意生成 ⭐⭐⭐

Part A: Supervisor 模式

3. 模式概述

Supervisor 是多 Agent 最经典的模式：

                       ┌─────────────────────┐
                       │     Supervisor       │
                       │   (LLM 决策路由)     │
                       └──┬──────┬──────┬─────┘
                          │      │      │
                    ┌─────↓┐ ┌───↓──┐ ┌─↓──────┐
                    │ Code │ │Search│ │Customer│
                    │Agent │ │Agent │ │ Agent  │
                    └──┬───┘ └──┬───┘ └───┬────┘
                       │        │         │
                       └────────┼─────────┘
                                ↓
                       ┌─────────────────────┐
                       │    Supervisor 汇总   │
                       └─────────────────────┘

Supervisor 本身也是一个 LLM 调用——它不做事，只决策"下一个应该叫谁"。工作完一轮后 Supervisor 判断是「继续派人」还是「汇总结束」。

4. LangGraph 实现

from typing import TypedDict, Annotated, Literal
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langchain.chat_models import init_chat_model
from langgraph.prebuilt import create_react_agent

# ----- State -----
class TeamState(TypedDict):
    messages: Annotated[list, add_messages]
    next: str  # supervisor 决定下一个谁干活
    task: str  # 原始任务

# ----- 子 Agent 定义 -----
# 每个子 Agent 是一个独立的 create_react_agent（带专用 tool）

def code_agent_node(state: TeamState) -> dict:
    """Code Agent — 只负责写代码"""
    code_agent = create_react_agent(
        model,
        tools=[read_file, write_file, run_tests],
        system_prompt="你是代码专家。只处理和代码相关的任务。完成后用中文汇报结果。",
    )
    result = code_agent.invoke({"messages": state["messages"]})
    # 取最后一条消息作为给 supervisor 的汇报
    return {"messages": [result["messages"][-1]]}


def search_agent_node(state: TeamState) -> dict:
    """Search Agent — 只负责搜索和检索"""
    search_agent = create_react_agent(
        model,
        tools=[web_search, search_kb],
        system_prompt="你是搜索专家。只处理信息检索任务。完成后用中文汇报结果。",
    )
    result = search_agent.invoke({"messages": state["messages"]})
    return {"messages": [result["messages"][-1]]}


# ----- Supervisor：核心路由决策 -----
MEMBERS = ["code_agent", "search_agent"]

system_prompt = f"""你是团队监督者。团队成员：
- code_agent: 写代码、调试、运行测试
- search_agent: 搜索网页、查询知识库

基于对话历史和当前任务，决定下一步：
(1) 指派某个成员处理 → 输出其名称
(2) 如果任务已完成 → 输出 "FINISH"

只输出成员名称或 "FINISH"，不要其他内容。
"""

def supervisor_node(state: TeamState) -> dict:
    """Supervisor 决定路由"""
    messages = [
        {"role": "system", "content": system_prompt},
        *state["messages"],
    ]
    response = model.invoke(messages)
    decision = response.content.strip()
    return {"next": decision}


# ----- 路由函数 -----
def route_supervisor(state: TeamState) -> Literal[
    "code_agent", "search_agent", "__end__"
]:
    next_agent = state["next"]
    if next_agent == "FINISH":
        return "__end__"
    if next_agent in MEMBERS:
        return next_agent
    return "__end__"  # fallback


# ----- 构建 Graph -----
builder = StateGraph(TeamState)
builder.add_node("supervisor", supervisor_node)
builder.add_node("code_agent", code_agent_node)
builder.add_node("search_agent", search_agent_node)

builder.add_edge(START, "supervisor")

# 条件边：supervisor 决定谁去干活
builder.add_conditional_edges("supervisor", route_supervisor)

# 子 Agent 干完活 → 回到 supervisor（关键：形成循环）
builder.add_edge("code_agent", "supervisor")
builder.add_edge("search_agent", "supervisor")

graph = builder.compile()

# ----- 运行 -----
result = graph.invoke({
    "messages": [
        {"role": "user", "content": "帮我查一下 Python 3.14 的新特性，并写一段示例代码展示 match-case 用法"}
    ],
    "task": "查资料 + 写代码",
    "next": "",
})
# 工作流：supervisor → search_agent → supervisor → code_agent → supervisor → FINISH
面试要点：Supervisor 模式的核心是"决策-执行-汇报-再决策"循环。Supervisor 本身是一个轻量 LLM 调用（不带 tool），子 Agent 各自有专用 tool。这样每个 Agent 的上下文都很干净。

Part B: Subagent 委托模式

5. 模式概述

Subagent 模式是把子 Agent 包装成一个 Tool：

主 Agent（带 N 个工具）
  ├── tool: get_weather      ← 普通函数
  ├── tool: search_kb        ← 普通函数
  ├── tool: code_subagent   ← 一个完整的子 Agent！
  └── tool: research_subagent ← 另一个子 Agent！

主 Agent 决定「这件事自己不干，委托给专业的子 Agent 干」。子 Agent 可以有自己独立的 tool 集合和 system prompt。

对比 Supervisor 模式：

SupervisorSubagent
控制权Supervisor 集中决策主 Agent LLM 选择委托
子 Agent 是Graph 中的一个 node主 Agent 的一个 tool
适用任务类型明确可分任务类型多变，需灵活委托
复杂度需要额外的图结构每层都是独立 Agent

6. 代码实现

from langchain.agents import create_agent
from langchain_core.tools import tool
from langgraph.prebuilt import create_react_agent

model = init_chat_model("openai:deepseek-chat", temperature=0, ...)

# Step 1: 定义子 Agent（作为 Tool 被主 Agent 调用）
@tool
def code_reviewer(code: str) -> str:
    """Review code for bugs, security issues, and style problems.

    Args:
        code: The source code to review (full file content)
    """
    reviewer = create_react_agent(
        model,
        tools=[],  # reviewer 不需要额外 tool
        system_prompt="""你是资深代码审查专家。审查代码时关注：
1. 逻辑错误和 Bug
2. 安全漏洞
3. 性能问题
4. 代码风格

用中文给出结构化反馈：(问题) + (严重程度) + (修复建议)。""",
    )
    result = reviewer.invoke({
        "messages": [{"role": "user", "content": f"审查以下代码：\n\n{code}"}]
    })
    return result["messages"][-1].content


@tool
def researcher(topic: str) -> str:
    """Research a topic thoroughly and return a structured summary.

    Args:
        topic: The topic to research
    """
    researcher_agent = create_react_agent(
        model,
        tools=[],  # 可加 web_search 等
        system_prompt="""你是研究专家。对任何话题：
1. 列出核心概念和定义
2. 对比不同方案/观点
3. 给出最佳实践建议
4. 用中文回答，结构清晰。""",
    )
    result = researcher_agent.invoke({
        "messages": [{"role": "user", "content": f"研究以下话题：{topic}"}]
    })
    return result["messages"][-1].content


# Step 2: 主 Agent — 带子 Agent 作为 tool
main_agent = create_agent(
    model=model,
    tools=[code_reviewer, researcher, get_weather],
    system_prompt="你是全栈工程师。可以写代码、审查代码、做技术调研。"
                   "交给子 Agent 处理专业任务。",
)

# 使用
result = main_agent.invoke({
    "messages": [{
        "role": "user",
        "content": "帮我研究一下 GraphQL vs REST 的优缺点，然后审查这段代码：def foo(): pass"
    }]
})
# 主 Agent 会：
# 1. 调用 researcher("GraphQL vs REST") → 获取研究报告
# 2. 调用 code_reviewer("def foo(): pass") → 获取审查意见
# 3. 汇总两个子 Agent 的结果给出最终回答
关键设计决策：子 Agent 的返回值应该是什么？推荐返回自然语言结果而不是结构化 JSON。这样主 Agent 可以直接理解子 Agent 的输出，不需要额外的解析层。面试能说清楚这个 tradeoff 是加分项。

Part C: Subgraph 嵌套模式

7. 用 Subgraph 实现可复用 Agent

LangGraph 支持把一个 Graph 作为另一个 Graph 的节点——这就是 Subgraph。适合构建可复用的 Agent 模块。

from langgraph.graph import StateGraph

# Step 1: 构建子 Graph（可复用的搜索 Agent）
def build_search_subgraph() -> StateGraph:
    """构建一个"搜索→评估→必要时再搜"的子图"""
    builder = StateGraph(SearchState)
    builder.add_node("search", search_node)
    builder.add_node("evaluate", evaluate_node)
    builder.add_node("rewrite", rewrite_query_node)
    builder.add_edge(START, "search")
    builder.add_conditional_edges("search", route_after_search)
    builder.add_edge("rewrite", "search")
    builder.add_edge("evaluate", END)
    return builder.compile()


# Step 2: 把子图嵌入父图
class MainState(TypedDict):
    messages: Annotated[list, add_messages]
    query: str
    search_output: str
    final_answer: str

main_builder = StateGraph(MainState)

# 子图作为节点 — 关键！
main_builder.add_node(
    "search_phase",
    build_search_subgraph(),  # ← 编译好的子图直接当节点用
)

main_builder.add_node("answer", answer_node)
main_builder.add_edge(START, "search_phase")
main_builder.add_edge("search_phase", "answer")
main_builder.add_edge("answer", END)

main_graph = main_builder.compile()

Subgraph 的优势：

  • 可复用：同一个子图可以被多个父图引用
  • 可测试：子图独立编译、独立测试
  • 关注分离：子图内部的复杂逻辑对外不可见，父图只需要知道输入输出

8. State 在父子图间如何传递？

面试高频细节题。默认行为：

  • 子图接收父图的完整 State（所以子图 State 的字段必须是父图 State 的子集）
  • 子图修改的字段会在子图执行完后写回父图
  • 子图内部的状态变更对父图不可见（黑盒封装）
# 父图 State 包含子图 State 的所有字段
class ParentState(TypedDict):
    query: str           # 父图独有
    messages: list       # 共享
    search_results: list # 共享（子图写入）

# 子图 State 是父图的子集
class ChildState(TypedDict):
    messages: list       # 从父图继承
    search_results: list # 子图写入，结束后写回父图

Part D: 面试核心 — 模式选择指南

9. 什么时候用什么模式？

场景推荐模式原因
客服：投诉/退货/咨询 Supervisor 路由 任务类型可预定义，按意图分流
技术研究+写作 Subagent 委托 主 Agent 灵活决策何时研究、何时写
代码生成+审查+测试 Supervisor + 顺序管道 三个阶段有严格先后顺序
创意头脑风暴 Swarm 协商 多视角并行，无正确错误之分
复杂审批工作流 Supervisor + Subgraph 每阶段内部复杂，阶段间 supervisor 调度
多个独立的简单工具调用 单 Agent（create_agent） 不需要多 Agent 的复杂度
面试反模式提醒：多 Agent 不是银弹。如果单 Agent + 5 个 tool 能解决的事，引入多 Agent 只会增加延迟和复杂度。面试时先说"先评估是否真的需要多 Agent"能展示工程判断力。

10. 面试高频追问

问题你的回答要点
多 Agent 怎么通信？ 两种方式：1) 通过 State（共享内存，子 Agent 写入，其他 Agent 读取）2) 通过 Supervisor 转发消息。更复杂的场景用消息队列（但不常见）。
多 Agent 的 token 消耗怎么控制？ 每个子 Agent 独立 context window。策略：1) 子 Agent 用短 system prompt + 专用 tool 2) Supervisor 只看到子 Agent 的输出摘要 3) 子 Agent 用更便宜的模型（如 haiku 做搜索，sonnet 做推理）。
怎么防止子 Agent 跑偏？ 1) 子 Agent system prompt 收窄职责范围 2) Supervisor 设超时/迭代上限 3) 子 Agent 返回结果时包含置信度 4) 加 evaluator 节点审核子 Agent 输出。
Supervisor 模式下怎么处理失败？ 子 Agent 不抛异常，返回错误描述。Supervisor 根据错误类型决策：换一个 Agent 重试、给原 Agent 更详细的指令、或向上汇报无法处理。这是"优雅降级"的设计模式。
和 OpenAI Swarm 的 handoff 有什么区别？ Swarm 的 handoff 是 Agent 主动交出控制权（"我不干了，你来"），LangGraph 的 Supervisor 是集中控制（"你去干活，干完回来汇报"）。集中控制更可控，handoff 更灵活。LangGraph 也支持类似 handoff 的 Command(goto=...) 模式。
