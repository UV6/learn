Lesson 0001: MCP 协议架构

2026-06-30 · 预计时间 35 分钟 · 难度：中级

本课目标：理解 MCP 解决了什么问题、协议的分层架构、以及 MCP 和其他工具集成方案（Function Calling, Plugin）的本质区别。

1. Before MCP：AI 工具集成的碎片化

在 MCP 之前，每个 LLM 平台有自己的工具定义格式：

OpenAI:    function_call + JSON Schema tools
Anthropic: tool_use + XML-style input_schema
Google:    function_declaration + OpenAPI-ish
LangChain: @tool decorator → 内部转换 → provider-specific format

问题：每写一次工具，要适配 N 种格式。没有统一的工具"语言"。

MCP 解决的是：定义一种开放的、跨平台的工具协议——Server 实现一次，所有 Client 都能用。

一句话：MCP 之于 AI 工具，就像 HTTP 之于 Web 服务——它是一个标准协议，让任何 AI 客户端都能发现和调用任何 MCP 服务器提供的能力。

2. MCP 的分层架构

┌─────────────────────────────────────────────────────┐ │ MCP Architecture │ │ │ │ ┌──────────────────────────────────────────────┐ │ │ │ Application Layer │ │ │ │ What you build: Tools, Resources, Prompts │ │ │ ├──────────────────────────────────────────────┤ │ │ │ Protocol Layer │ │ │ │ JSON-RPC 2.0 messages │ │ │ │ · tools/list · resources/read · prompts/get │ │ │ ├──────────────────────────────────────────────┤ │ │ │ Transport Layer │ │ │ │ · stdio (本地进程间通信) │ │ │ │ · HTTP + SSE (远程服务) │ │ │ └──────────────────────────────────────────────┘ │ │ │ │ Client (Claude Desktop/Code, Cursor, ...) │ │ ↕ JSON-RPC over stdio/HTTP │ │ Server (your code: 数据库查询、API 封装、文件操作) │ └─────────────────────────────────────────────────────┘
层做什么你接触什么
Application 定义 Server 提供的能力 Tools（可调用的函数）、Resources（可读取的数据）、Prompts（预设提示词模板）
Protocol Client 和 Server 之间怎么对话 JSON-RPC 2.0 格式的 request/response/notification
Transport 消息怎么传输 stdio（本地）或 HTTP+SSE（远程）

3. MCP 的三种能力：Tools、Resources、Prompts

类型方向类比示例
Tools Client → Server (调用) RPC / function call search_code(query)、create_issue(title, body)
Resources Client ← Server (读取) REST GET file://project/README.md、db://users/schema
Prompts Client ← Server (模板) 预置 prompt code-review 触发标准的代码审查 prompt
关键区分：Tool 是"模型主动决定调用"（模型说"我需要搜索结果"），Resource 是"模型被动读取"（模型说"给我看一下那个文件"）。前者是行动，后者是数据。Prompt 是两者的粘合剂——预设的场景模板。

4. JSON-RPC 2.0：MCP 的通信语言

MCP 的所有消息都是 JSON-RPC 2.0 格式。我们只需要了解三种消息类型：

// 1. Request — Client 发给 Server
{"jsonrpc": "2.0", "id": 1, "method": "tools/call",
 "params": {"name": "get_weather", "arguments": {"city": "Beijing"}}}

// 2. Response — Server 返回给 Client
{"jsonrpc": "2.0", "id": 1,
 "result": {"content": [{"type": "text", "text": "晴朗，25°C"}]}}

// 3. Notification — 单向消息（不需要回复）
{"jsonrpc": "2.0", "method": "notifications/initialized"}

Protocol 层定义的核心方法（面试必知）：

方法类型用途
initializeRequestClient 连接 Server，交换能力信息
tools/listRequest获取 Server 提供的所有 Tool
tools/callRequest调用一个具体的 Tool
resources/readRequest读取一个 Resource 的内容
resources/listRequest列出所有可用 Resource
prompts/getRequest获取一个 Prompt 模板
notifications/tools/list_changedNotificationServer 通知 Client 工具列表变了

5. Transport 层：stdio vs HTTP

stdioHTTP + SSE
适用本地工具（Claude Code, VS Code 插件）远程服务、Web 应用
启动方式Client 启动 Server 进程Server 作为 HTTP 服务运行
通信stdin/stdout 管道HTTP POST + Server-Sent Events
配置一行命令需要 URL + 可选认证
延迟极低（本地进程）网络延迟

大多数开发场景用 stdio——在你的 claude_desktop_config.json 或 .mcp.json 中配一条命令就行。下节课会实战。

6. MCP vs Function Calling vs Plugin：三者的关系

MCPFunction CallingPlugin
是什么开放协议LLM 的一个特性平台生态
定义者Anthropic（开源）各 LLM 厂商各平台
跨平台✅ 是（目标）❌ 格式不统一❌ 平台绑定
发现机制tools/list 自动发现手动在 API 调用中传入平台商店安装
传输stdio + HTTPHTTP API平台内部
权限用户批准依赖平台安装时授权
面试金句："MCP 和 Function Calling 不在同一层。Function Calling 是 LLM 判断'要不要调工具'的能力，MCP 是工具'如何被发现和调用'的协议。MCP Server 的工具最终通过 Function Calling 机制被 LLM 调用——它们是协作关系，不是替代关系。"

7. 面试高频追问

问题你的回答要点
"为什么不直接用 REST API？" REST API 需要 LLM 理解 URL 结构、HTTP 方法、认证方式——每个 API 都是独特的。MCP 统一了工具的描述格式（JSON Schema），LLM 只需要理解一种"语言"。而且 MCP 的 tools/list 提供了自动发现，REST API 需要手动写文档。
"MCP 的安全性怎么保证？" 三层：(1) Transport 层隔离——stdio 天然进程隔离，只有 localhost (2) 用户批准——每次工具调用都需要用户确认（类似 sudo）(3) Server 端权限——Server 自己实现访问控制。Claude Code 目前所有工具调用都需要用户授权。
"MCP 的局限性？" (1) 协议还在演进，生态不够成熟 (2) stdio transport 不适合分布式场景 (3) 没有内置的认证/鉴权标准 (4) 大文件传输需要走 Resource 而非 Tool 返回值 (5) 目前主要是文本输入输出，二进制/流式支持有限。
"你怎么看待 MCP 的竞争？" OpenAI 的 Plugins 已停止发展，Google 的 A2A (Agent-to-Agent) 协议聚焦 Agent 间通信而非工具调用——MCP 和 A2A 是互补的。MCP 作为开源标准有先发优势和社区采纳（Claude, Cursor, Zed, Sourcegraph 等都已支持）。
