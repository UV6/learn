Lesson 0004: Claude Code Skill 开发

2026-06-30 · 预计时间 40 分钟 · 难度：中高级

本课目标：理解 Skill 的构成（YAML frontmatter + Markdown body + hooks），能独立开发一个可发布的自定义 Skill。

1. Skill 的构成

一个 Skill 就是一个 Markdown 文件（.md），放在 ~/.claude/skills/ 目录（用户级）或项目的 .claude/skills/ 目录（项目级）。

---                                                                  ← YAML frontmatter
name: my-skill-name
description: 一行描述，用于匹配触发条件
allowed-tools: Bash, Read, Write, Edit
metadata:
  category: development
---

# Skill 标题                                                        ← Markdown body

## Instructions

你被激活后应当遵循的指令...

2. Skill 的触发机制

Claude Code 通过 skill description 匹配来激活 Skill——不需要用户显式调用 /skill-name（虽然也可以）。当用户的消息与某个 Skill 的描述匹配时，Skill 被自动加载。

这意味着 description 是 Skill 最重要的部分。它决定了 Skill 何时被激活。

Description 写法激活场景
Use when the user wants to deploy to production"部署到线上"、"发布"、"ship it"
Use when user needs to search, create, or manage notes in Obsidian"帮我记个笔记"、"查一下之前的笔记"
Use when debugging hard bugs with systematic diagnosis loop"debug一下"、"这有个bug"、"帮我排查"

description 要宽到能触发，窄到不误触发。这是 Skill 设计中最难的部分。

3. 实战：写一个 Skill

场景：创建一个 Skill，让 Claude Code 在修改任何 Python 文件后自动运行测试。

创建 .claude/skills/auto-test.md：

---
name: auto-test
description: Run pytest automatically after modifying Python files. Use when user edits .py files or asks to run tests.
allowed-tools: Bash(pytest:*), Read
hooks:
  PostToolUse:
    - matcher: Edit|Write
      paths: ["*.py"]
      hooks:
        - command: ${CLAUDE_PLUGIN_ROOT}/hooks/run-tests.sh
---

# Auto Test Skill

## When Activated

After any Write or Edit to a Python file, run the project's test suite.

## Behavior

1. Detect which test file matches the changed source file
2. Run `pytest  -x -q`
3. Report results to the user

## Configuration

Set `TEST_COMMAND` env var to override the test runner (default: pytest).

创建 .claude/skills/auto-test/hooks/run-tests.sh：

#!/bin/bash
# run-tests.sh — triggered after Python file edits
set -e

# Find matching test file
CHANGED_FILE="$1"
if [[ "$CHANGED_FILE" == *.py ]]; then
    TEST_FILE="tests/test_$(basename $CHANGED_FILE)"
    if [ -f "$TEST_FILE" ]; then
        echo "→ 运行测试: $TEST_FILE"
        python -m pytest "$TEST_FILE" -x -q 2>&1
    else
        echo "⚠️  未找到对应的测试文件: $TEST_FILE"
    fi
fi

4. Skill 的钩子系统

Hook 类型触发时机典型用途
PreToolUse 工具调用前 权限检查、参数验证
PostToolUse 工具调用后 自动测试、格式化、通知
Notification 特定事件发生时 长时间任务完成通知
UserPromptSubmit 用户提交消息时 注入上下文、前置检查
SessionStart Session 开始时 加载项目状态、环境检查
Stop Agent 停止时 清理资源、保存状态
Skill 和 MCP 的关系：Skill 是给Claude 的指令和行为规则——"遇到这种情况怎么做"。MCP 是 Claude 的外部能力——"给你一个工具去调用"。Skill 可以调用 MCP 工具，MCP 工具可以被 Skill 使用。它们互补，不在同一层。

5. Skill 的最佳实践

  • Description 精心设计：包含触发词、场景、排除条件。写完后测试 5 个触发句子和 5 个不应该触发的句子。
  • Skill 要短：正文控制在 200 行以内。Skill 加载到 context window，太长会挤占用户对话空间。
  • 分层设计：工具类 Skill (MCP Server) + 行为类 Skill (system prompt) + 编排类 Skill (hook)。不要让一个 Skill 做所有事。
  • 渐进式指令：从"必须做"→"应该做"→"可以做"，用 MUST/SHOULD/MAY 标注优先级。

6. Skill 目录结构参考

参考 superpowers 的 Skill 组织方式：

7. 面试高频追问

问题你的回答要点
"Skill 和 MCP 什么关系？" Skill 定义行为和流程（"怎么做"），MCP 提供工具和数据（"用什么做"）。一个 Skill 可以调用多个 MCP Server 的工具。类比：Skill = playbook，MCP = toolkit。
"Skill 的权限模型？" Skill 通过 allowed-tools 声明所需权限——只授予它需要的，不给多余的。这是最小权限原则。Hook 脚本独立运行，权限取决于脚本本身。
"怎么测试 Skill？" 1) 单元测试 hook 脚本 2) 用 eval prompt 验证 description 的触发准确性 3) 实际场景走查——触发前/中/后的完整流程。
